# 校园拼单平台冻结版开发契约

本文档作为当前仓库的正式开发基线，用于统一前后端接口命名、数据结构、状态流转、定时任务和测试验收口径。

适用范围：

- 前端：`frontend`
- 后端：`backend`
- 当前阶段：课程项目第一版，优先打通主业务闭环

统一原则：

- 前端统一使用 `axios`
- 接口统一前缀为 `/api`
- 统一返回结构为 `{ code, message, data }`
- 普通用户与管理员分离登录
- 所有状态名、接口名、字段名只保留一套
- V1 不引入 MQ，不做通知中心页面，不做真实支付

## 1. 统一约定

| 项目 | 最终版 |
| --- | --- |
| 前端请求库 | `axios` |
| 接口前缀 | `/api` |
| 鉴权头 | `Authorization: Bearer <token>` |
| 成功响应 | `{ "code": 0, "message": "success", "data": ... }` |
| 失败响应 | `{ "code": 非0, "message": "错误信息", "data": null }` |
| 默认时间格式 | `yyyy-MM-dd HH:mm:ss` |
| 金额类型 | 前端 JSON `number`，后端 `BigDecimal(10,2)` |
| 分页结构 | `{ "list": [], "page": 1, "pageSize": 10, "total": 0, "pages": 0 }` |
| 角色枚举 | `USER` / `ADMIN` |
| 用户状态 | `NORMAL` / `BANNED` |
| 订单主状态 | `OPEN` / `GROUPED` / `WAIT_DELIVERY` / `WAIT_RECEIVE` / `COMPLETED` / `CANCELED` |

## 6. 状态机冻结规则

### 6.1 订单主状态

| 状态 | 进入条件 | 允许动作 | 退出条件 | 副作用 |
| --- | --- | --- | --- | --- |
| `OPEN` | 发起人创建拼单成功 | `join`, `pay`, `exit` | 满员且全部 `ACTIVE` 成员 `PAID` 进入 `GROUPED`；超时未成团进入 `CANCELED` | 创建发起人成员行，发起人也必须支付 |
| `GROUPED` | 满员且全部 `ACTIVE` 成员 `PAID` | `upload-receipt` | 上传凭证进入 `WAIT_DELIVERY`；超时未上传凭证只开放投诉，不直接取消 | 写 `receiptUploadDeadlineAt` |
| `WAIT_DELIVERY` | 发起人上传凭证 | `mark-delivered` | 发起人确认送达进入 `WAIT_RECEIVE`；超时只开放投诉，不直接取消 | 可能触发差额退款 |
| `WAIT_RECEIVE` | 发起人确认送达 | 参与成员 `confirm-received` | 全部 `ACTIVE` 成员都已收货进入 `COMPLETED` | 自动确认收货后也会尝试完成订单 |
| `COMPLETED` | 全部 `ACTIVE` 成员已收货 | 无 | 终态 | 给发起人结算，写完成日志 |
| `CANCELED` | 未成团超时 / 管理员取消 / 投诉成立 | 无 | 终态 | 对 `ACTIVE` 已支付成员净退款 |

### 6.2 成员支付状态

| 状态 | 含义 |
| --- | --- |
| `UNPAID` | 未支付 |
| `PAID` | 已支付 |

### 6.3 成员收货状态

| 状态 | 含义 |
| --- | --- |
| `NOT_READY` | 尚未进入收货阶段 |
| `WAIT_CONFIRM` | 待成员确认收货 |
| `RECEIVED` | 成员主动确认收货 |
| `AUTO_RECEIVED` | 系统自动确认收货 |

### 6.4 关键拍板

- 发起人也必须支付
- 只有 `currentMemberCount == totalMemberCount` 且全部 `ACTIVE` 成员 `payStatus=PAID` 才算成团
- 差额退款发生在上传凭证时，不改变主状态
- 投诉成立后订单固定进入 `CANCELED`
- 自动确认收货后立即调用 `tryCompleteOrder(orderId)` 判断是否完成
- 自动确认收货时间统一为 `30 分钟`
- 成团后 `30 分钟` 未上传凭证只开放投诉，不直接取消
- 超过预计最晚送达时间 `60 分钟` 只开放投诉，不直接取消

## 7. 状态转移边

| 转移 | 触发者 | 触发方式 | 条件 | 结果 |
| --- | --- | --- | --- | --- |
| `OPEN -> GROUPED` | 用户动作 | `POST /api/orders/{orderId}/pay` | 满员且全部 `ACTIVE` 成员已支付 | 订单成团，写上传凭证截止时间 |
| `OPEN -> CANCELED` | 系统任务 | `cancelExpiredOpenOrders()` | `deadlineAt <= now` 且未成团 | 订单取消，净退款 |
| `GROUPED -> WAIT_DELIVERY` | 用户动作 | `POST /api/orders/{orderId}/upload-receipt` | 发起人本人且状态为 `GROUPED` | 写凭证、写实际金额、必要时差额退款 |
| `WAIT_DELIVERY -> WAIT_RECEIVE` | 用户动作 | `POST /api/orders/{orderId}/mark-delivered` | 发起人本人且状态为 `WAIT_DELIVERY` | 写送达时间，更新成员收货状态 |
| `WAIT_RECEIVE -> COMPLETED` | 用户动作 / 系统任务 | `confirm-received` 或 `tryCompleteOrder()` | 全部 `ACTIVE` 成员已收货 | 订单完成并结算 |
| `任意进行中 -> CANCELED` | 管理员动作 | `POST /api/admin/orders/{orderId}/cancel` | 订单状态为进行中 | 订单取消，净退款 |
| `任意进行中 -> CANCELED` | 管理员动作 | `POST /api/admin/complaints/{complaintId}/handle` | 投诉成立 | 订单取消，净退款，发起人扣分 |

## 9. 定时任务与自动规则

统一实现方式：

- 使用 `@Scheduled + Service + @Transactional`
- 扫描频率统一先按 `1 分钟`
- 不引入 MQ，不做事件总线

### 9.1 任务表

| 任务名 | 扫描条件 | 频率 | 主要修改 | 流水 | 日志 | 通知 |
| --- | --- | --- | --- | --- | --- | --- |
| `cancelExpiredOpenOrders()` | `status=OPEN and deadline_at <= now` | 每 1 分钟 | `group_order.status=CANCELED`，`cancel_reason=DEADLINE_NOT_GROUPED`，`ACTIVE -> CANCELED` | 对 `ACTIVE && PAID` 生成 `REFUND_CANCEL` | `ORDER_AUTO_CANCELED` | 通知发起人和 `ACTIVE` 成员 |
| `openReceiptTimeoutComplaints()` | `status=GROUPED and complaint_opened=false and receipt_upload_deadline_at <= now and no order_receipt` | 每 1 分钟 | `group_order.complaint_opened=true` | 无 | `COMPLAINT_CHANNEL_OPENED_RECEIPT_TIMEOUT` | 通知非发起人 `ACTIVE` 成员 |
| `openDeliveryTimeoutComplaints()` | `status=WAIT_DELIVERY and complaint_opened=false and delivered_at is null and expected_delivery_end_at + 60m <= now` | 每 1 分钟 | `group_order.complaint_opened=true` | 无 | `COMPLAINT_CHANNEL_OPENED_DELIVERY_TIMEOUT` | 通知非发起人 `ACTIVE` 成员 |
| `autoConfirmReceivedMembers()` | `status=WAIT_RECEIVE and receive_status=WAIT_CONFIRM and delivered_at + 30m <= now` | 每 1 分钟 | 成员改为 `AUTO_RECEIVED`，写 `received_at` | 无 | `MEMBER_AUTO_RECEIVED` | 可选通知发起人 |
| `tryCompleteOrder(orderId)` | 所有 `ACTIVE` 成员均已收货 | 同步触发 + 定时兜底 | `group_order.status=COMPLETED` | 生成 `SETTLE_TO_CREATOR` | `ORDER_COMPLETED` | 通知发起人和成员 |

### 9.2 非定时但必须同步事务执行

| 规则 | 触发点 | 说明 |
| --- | --- | --- |
| 差额退款 | `upload-receipt` | `actualTotalAmount < estimatedTotalAmount` 时同步生成 `REFUND_DIFF` |
| 成团判断 | `pay` | 最后一名成员支付成功后立即判断是否进入 `GROUPED` |
| 完成判断 | `confirm-received` / `autoConfirmReceivedMembers()` | 完成收货后立即尝试完结订单 |

### 9.3 资金规则

- 取消退款金额：`refundCancelAmount = max(payAmount - refundAmountTotal, 0)`
- 差额退款金额：`(estimatedTotalAmount - actualTotalAmount) / totalMemberCount`
- 发起人结算金额：`settleToCreatorAmount = actualTotalAmount`
- 所有退款均按“净退款”执行，不允许重复退款
