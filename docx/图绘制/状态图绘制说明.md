# 状态图绘制说明

## 1. 绘图目标

状态图用于描述系统中核心业务对象在生命周期内的状态变化过程。
本系统状态图的重点对象包括：

1.  **拼单主对象（GroupOrder）**
2.  **拼单成员对象（GroupOrderMember）**

状态图的作用是回答以下问题：

-   一个拼单会经历哪些状态；
-   每个状态在什么条件下进入、在什么条件下退出；
-   哪些状态是终态；
-   成员在拼单中的参与状态如何变化；
-   自动化处理逻辑如何影响状态迁移。

状态图不用于描述对象之间的消息交互，也不用于描述数据库关系，因此不能把状态图画成顺序图或 ER 图。

------

## 2. 绘图总体要求

### 2.1 图的数量

本系统状态图建议绘制 **两张图**：

1.  **拼单主状态图**
2.  **拼单成员状态图**

这两张图分别对应系统中两个最关键的状态对象：

-   `GroupOrder`
-   `GroupOrderMember`

### 2.2 绘图方式

状态图采用标准 UML 状态图画法，必须包含：

-   初始状态（实心圆）
-   状态节点（圆角矩形）
-   状态迁移箭头
-   迁移条件说明
-   终态（如确有必要，可用同心圆，也可直接使用终态状态节点）

### 2.3 图中文字要求

图中的状态名称必须使用与正文一致的命名，不能自行更改。
迁移条件应写在箭头上，简洁明了，不要写成长句。

------

## 3. 拼单主状态图

## 3.1 绘图目标

拼单主状态图用于描述一条拼单从创建到完成或取消的完整生命周期。
系统中的拼单主状态统一采用以下六种状态：

-   `OPEN`
-   `GROUPED`
-   `WAIT_DELIVERY`
-   `WAIT_RECEIVE`
-   `COMPLETED`
-   `CANCELED`

这六种状态已经在概要设计、详细设计正文和数据库设计中保持统一，因此状态图中不得再新增其他主状态名称。拼单状态的中文含义分别为：拼单中、已成团、待送达、待收货、已完成、已取消。

------

## 3.2 状态定义

### （1）OPEN

表示拼单已发布，正在等待成员加入和完成支付。

进入条件：

-   发起人创建拼单成功

退出条件：

-   人数满员且全部成员支付完成
-   截止时间到仍未成团

### （2）GROUPED

表示拼单已经成团，等待发起人上传订单凭证。

进入条件：

-   当前人数达到总人数，且全部成员支付完成

退出条件：

-   发起人上传订单凭证
-   成团后 30 分钟内未上传凭证且投诉处理成立

### （3）WAIT_DELIVERY

表示发起人已上传订单凭证和实际金额信息，等待商品送达取货点。

进入条件：

-   发起人上传凭证成功

退出条件：

-   发起人点击“商品已送达”
-   超过预计最晚送达时间 60 分钟仍未送达且投诉处理成立

### （4）WAIT_RECEIVE

表示商品已送达，等待参与者确认收货。

进入条件：

-   发起人点击“商品已送达”

退出条件：

-   所有成员确认收货或系统自动确认收货

### （5）COMPLETED

表示拼单已正常完成，所有成员均已完成收货确认，平台可结算给发起人。

进入条件：

-   全体成员均已收货（包括主动确认收货和自动确认收货）

退出条件：

-   无

### （6）CANCELED

表示拼单已提前终止，不再继续执行后续流程。

进入条件：

-   截止时间到仍未成团
-   成团后未按时上传凭证且投诉处理成立
-   超时未送达且投诉处理成立
-   管理员强制取消

退出条件：

-   无

这些状态与需求文档中的拼单流程完全对应：拼单发布后进入拼单中，满足人数和支付条件后成团，上传凭证后进入待送达，确认送达后进入待收货，全员收货后完成；超时未成团、投诉成立或后台强制干预时取消。

------

## 3.3 拼单主状态迁移关系

绘图时应严格按以下迁移关系连接：

1.  初始状态 → `OPEN`
2.  `OPEN` → `GROUPED`
    -   条件：`人数满员 且 全员已支付`
3.  `OPEN` → `CANCELED`
    -   条件：`截止未成团`
4.  `GROUPED` → `WAIT_DELIVERY`
    -   条件：`上传凭证`
5.  `GROUPED` → `CANCELED`
    -   条件：`30分钟未上传凭证 且 投诉处理成立`
6.  `WAIT_DELIVERY` → `WAIT_RECEIVE`
    -   条件：`确认送达`
7.  `WAIT_DELIVERY` → `CANCELED`
    -   条件：`超时未送达 且 投诉处理成立`
8.  `WAIT_RECEIVE` → `COMPLETED`
    -   条件：`全员收货 / 自动确认完成`

### 需要注意的地方

-   `GROUPED` 不直接进入 `COMPLETED`
-   `WAIT_DELIVERY` 不直接进入 `COMPLETED`
-   `COMPLETED` 和 `CANCELED` 都是终态
-   若图中需要画终态节点，可在 `COMPLETED` 和 `CANCELED` 后分别接终态；如果图面拥挤，也可直接把这两个状态作为最终状态，不额外画终态圆圈

------

## 3.4 拼单主状态图中必须体现的规则

这张图必须体现以下规则：

1.  成团条件是**人数满员且全员支付**
2.  拼单截止未成团时直接取消
3.  成团后发起人必须上传凭证，否则不能进入后续流程
4.  上传凭证后进入待送达，而不是直接进入待收货
5.  发起人必须先确认送达，参与者才会进入确认收货阶段
6.  拼单完成的条件是所有成员已收货
7.  拼单取消不仅可能来自未成团，还可能来自投诉处理成立或管理员干预

这些规则都属于系统的核心业务约束，状态图中必须明确表现。

------

## 4. 拼单成员状态图

## 4.1 绘图目标

拼单成员状态图用于描述单个成员在某个拼单中的参与生命周期。

需要说明的是，系统实际实现中成员状态由三类字段共同表达：

-   `joinStatus`
-   `payStatus`
-   `receiveStatus`

但在状态图中，为了便于整体展示成员参与过程，可以将成员状态抽象成一条完整生命周期状态链，用于概括业务过程。这样的表达方式与数据库字段拆分并不冲突，前者服务于设计展示，后者服务于具体实现。系统设计稿中已经对这三类状态字段进行了详细划分。

------

## 4.2 成员状态定义

建议成员状态图使用以下状态：

-   `UNPAID`
-   `PAID`
-   `WAIT_CONFIRM`
-   `RECEIVED`
-   `AUTO_RECEIVED`
-   `EXITED`
-   `CANCELED`

### （1）UNPAID

表示成员已加入拼单，但尚未完成支付。

进入条件：

-   用户加入拼单成功

退出条件：

-   完成支付
-   主动退出

### （2）PAID

表示成员已完成支付，正在等待后续流程推进。

进入条件：

-   用户支付成功

退出条件：

-   拼单取消
-   发起人确认送达后进入待确认收货

### （3）WAIT_CONFIRM

表示拼单已进入待收货阶段，当前成员等待确认收货。

进入条件：

-   发起人点击“商品已送达”

退出条件：

-   用户主动确认收货
-   系统自动确认收货

### （4）RECEIVED

表示成员主动点击确认收货，完成个人收货过程。

进入条件：

-   用户点击确认收货

退出条件：

-   无

### （5）AUTO_RECEIVED

表示成员未在规定时间内确认收货，由系统自动确认。

进入条件：

-   确认送达后 30 分钟内未确认收货

退出条件：

-   无

### （6）EXITED

表示成员在拼单中阶段主动退出拼单。

进入条件：

-   用户在允许退出阶段主动退出

退出条件：

-   无

### （7）CANCELED

表示由于拼单取消，成员参与关系被动结束。

进入条件：

-   拼单取消

退出条件：

-   无

------

## 4.3 成员状态迁移关系

绘图时应严格按以下迁移关系连接：

1.  初始状态 → `UNPAID`
2.  `UNPAID` → `PAID`
    -   条件：`支付成功`
3.  `UNPAID` → `EXITED`
    -   条件：`主动退出`
4.  `PAID` → `CANCELED`
    -   条件：`拼单取消`
5.  `PAID` → `WAIT_CONFIRM`
    -   条件：`发起人确认送达`
6.  `WAIT_CONFIRM` → `RECEIVED`
    -   条件：`确认收货`
7.  `WAIT_CONFIRM` → `AUTO_RECEIVED`
    -   条件：`30分钟未确认`

如果绘图组员希望再细化，也可以补一条：

-   `PAID` → `EXITED`
    -   条件：`拼单仍处于 OPEN 且 用户主动退出`

但如果图面已经清晰，也可以保留在 `UNPAID` / `PAID` 上方统一写说明：“在拼单中阶段允许退出”，不用单独再拉两条线。
不过从完整性来看，更推荐把 `PAID → EXITED` 也画出来，因为需求中明确规定：未成团时可退出并自动退款，已成团禁止退出。

------

## 4.4 成员状态图中必须体现的规则

这张图必须体现以下规则：

1.  成员加入后不是直接已支付，而是先进入 `UNPAID`
2.  只有支付成功后才进入 `PAID`
3.  只有在允许阶段才可以主动退出
4.  拼单取消后，成员状态应转为 `CANCELED`
5.  发起人确认送达后，成员才进入待确认收货状态
6.  用户可主动确认收货，也可能由系统自动确认
7.  `RECEIVED` 和 `AUTO_RECEIVED` 都是终态

需求中明确规定：未成团状态下成员可退出并自动退款；已成团后禁止退出；发起人确认送达后参与者取货，若 30 分钟内未点击确认收货，则系统默认完成收货。

------

## 5. 状态图推荐版式

### 5.1 拼单主状态图版式

建议采用横向或环形简单布局，推荐如下顺序：

```
OPEN → GROUPED → WAIT_DELIVERY → WAIT_RECEIVE → COMPLETED
```

然后将 `CANCELED` 放在下方或右下角，分别从：

-   `OPEN`
-   `GROUPED`
-   `WAIT_DELIVERY`

拉箭头连过去。

这种版式可以清楚地表现：

-   上方主线是正常流程
-   下方分支是异常终止流程

### 5.2 成员状态图版式

建议采用横向主线布局：

```
UNPAID → PAID → WAIT_CONFIRM → RECEIVED
```

然后：

-   从 `WAIT_CONFIRM` 向下拉到 `AUTO_RECEIVED`
-   从 `UNPAID` 和 `PAID` 向下拉到 `EXITED`
-   从 `PAID` 向右下拉到 `CANCELED`

这样画可以清楚表现：

-   正常路径是支付 → 待确认 → 收货
-   异常路径是退出或取消
-   自动路径是自动确认收货

------

## 6. 图中命名统一要求

### 6.1 拼单主状态必须统一为

-   `OPEN`
-   `GROUPED`
-   `WAIT_DELIVERY`
-   `WAIT_RECEIVE`
-   `COMPLETED`
-   `CANCELED`

### 6.2 成员状态必须统一为

-   `UNPAID`
-   `PAID`
-   `WAIT_CONFIRM`
-   `RECEIVED`
-   `AUTO_RECEIVED`
-   `EXITED`
-   `CANCELED`

### 6.3 不要混用的命名

不要在图中出现以下混乱命名：

-   `RECRUITING` / `OPEN` 混用
-   `WAITING_DELIVERY` / `WAIT_DELIVERY` 混用
-   `WAITING_RECEIPT_CONFIRM` / `WAIT_RECEIVE` 混用
-   `RESOLVED` / `PROCESSED` 混用在投诉图中
-   `REFUNDED` 作为成员主状态直接替代 `EXITED` 或 `CANCELED`

状态图里使用的状态名称必须和你们这份详细设计正文一致，不可自行更改。

## 8. 图下注释建议

### 拼单主状态图图注

本图展示校园拼单平台中拼单对象从创建、成团、上传凭证、确认送达、确认收货到完成或取消的完整状态迁移过程。

### 成员状态图图注

本图展示校园拼单平台中成员对象在单个拼单中的参与生命周期，包括加入、支付、退出、待确认收货、主动确认收货和自动确认收货等状态变化过程。

------

## 9. 组员绘图时的注意事项

1.  状态图只画状态和迁移，不画对象之间的消息调用。
2.  所有迁移箭头上都要写条件，不要只画箭头不写原因。
3.  拼单主状态图一定要把 `CANCELED` 画成异常终态。
4.  成员状态图一定要体现 `AUTO_RECEIVED`。
5.  成员状态图不要直接照搬数据库字段名称 `joinStatus / payStatus / receiveStatus` 作为状态节点名称。
6.  图中不要引入正文没有的状态名称。
7.  中文说明可以放在图注中，状态节点本身统一写英文。
8.  若图面拥挤，可不画终态双圆，而直接把 `COMPLETED`、`CANCELED`、`RECEIVED`、`AUTO_RECEIVED`、`EXITED` 作为终态即可。

------
