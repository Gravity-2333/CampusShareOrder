# 顺序图与协作图绘制说明

## 1. 绘图目标

顺序图与协作图用于描述系统在关键业务场景下，各参与对象之间的消息交互关系。

其中：

-   **顺序图**强调消息发生的**时间先后顺序**
-   **协作图**强调对象之间的**通信关系与职责分工**

这两类图都属于动态模型，用于补充类图和状态图无法直接表达的“业务过程”。

本系统的动态图重点围绕两类业务展开：

1.  **正常业务闭环**：创建拼单、加入拼单、支付、成团、上传凭证、确认送达、确认收货、完成拼单
2.  **异常业务闭环**：超时未上传凭证、超时未送达、投诉提交、管理员处理、退款与扣分

因此，本次详细设计中建议绘制：

1.  发起拼单并成团顺序图
2.  上传凭证到确认收货完成顺序图
3.  投诉提交与管理员处理顺序图
4.  拼单主流程协作图

------

## 2. 绘图总体要求

### 2.1 顺序图要求

顺序图采用 UML 顺序图标准画法，必须包含以下元素：

-   参与者（Actor）
-   对象（Object）
-   生命线（Lifeline）
-   激活条（Activation）
-   消息箭头（Message）
-   返回消息（可省略部分简单返回）

顺序图必须从左到右排列参与对象，从上到下表示时间顺序。

### 2.2 协作图要求

协作图采用 UML 协作图（通信图）标准画法，必须包含以下元素：

-   参与对象
-   对象之间的连接线
-   消息编号
-   消息名称

协作图不强调纵向时间轴，而强调“谁和谁通信”“消息由谁发给谁”。

### 2.3 不允许出现的问题

动态图中不要出现以下问题：

1.  不要把顺序图画成流程图
2.  不要把协作图画成类图
3.  不要在图中使用与正文不一致的类名或模块名
4.  不要引入正文中未出现的服务组件，例如“支付中心”“推荐服务”“消息队列”
5.  不要把数据库表名直接当成对象名写进图里
6.  不要把业务状态写乱，例如把 `OPEN` 写成“招募中”但图中又没有英文统一口径

------

### 3.1 参与者命名统一

-   发起人

-   管理员
-   系统定时任务（如确需单独表现）

### 3.2 前后端对象命名统一

-   前端页面
-   AuthController
-   UserController
-   OrderController
-   ComplaintController
-   AdminController
-   AuthService
-   UserService
-   OrderService
-   ComplaintService
-   AdminService

### 3.3 核心实体对象命名统一

-   GroupOrder
-   GroupOrderMember
-   OrderReceipt
-   Complaint
-   CapitalRecord

### 3.4 状态名统一

拼单主状态统一使用：

-   `OPEN`
-   `GROUPED`

-   `PENDING`
-   `PROCESSED`

不要在同一张图里混用“已成团 / GROUPED”“待送达 / WAIT_DELIVERY”两套写法。
正文写的是英文状态名时，图中也统一使用英文状态名；中文解释可以写在图注或箭头说明中。

## 4. 顺序图一：发起拼单并成团

## 4.1 绘图目标

该顺序图用于描述从发起人创建拼单，到参与者加入并支付，最终系统自动成团的全过程。

它要解决的问题是：

-   发起人如何创建拼单
-   发起人为什么也算成员之一
-   参与者如何加入并支付
-   系统在什么条件下自动成团
-   成团后如何通知相关用户

------

## 4.2 参与对象

建议从左到右排列如下：

1.  发起人
2.  参与者
3.  前端页面
4.  OrderController
5.  OrderService
6.  GroupOrder
7.  GroupOrderMember
8.  CapitalRecord
9.  NotificationService

如果图面太挤，可以不把 `GroupOrder` 和 `GroupOrderMember` 作为独立对象分开画，而是由 `OrderService` 直接操作；但为了体现业务对象，建议保留。

------

## 4.3 主要消息流程

绘图时按以下顺序表达：

### 第一阶段：发起人创建拼单

1.  发起人 → 前端页面：填写拼单信息并提交
2.  前端页面 → OrderController：`createOrder(orderForm)`
3.  OrderController → OrderService：`createOrder(orderForm, currentUser)`
4.  OrderService → GroupOrder：创建拼单对象
5.  OrderService → GroupOrderMember：创建发起人成员记录

这里要在图中体现：
**发起人发布拼单后，系统自动将其加入拼单成员列表。**

### 第二阶段：发起人完成支付

1.  发起人 → 前端页面：发起支付
2.  前端页面 → OrderController：`payOrder(orderId)`
3.  OrderController → OrderService：`payOrder(orderId, currentUser)`
4.  OrderService → GroupOrderMember：更新发起人支付状态
5.  OrderService → CapitalRecord：记录支付流水

这里要体现：
**发起人作为参与者之一，也必须支付预估人均金额。**

### 第三阶段：参与者加入并支付

1.  参与者 → 前端页面：查看拼单详情并点击加入
2.  前端页面 → OrderController：`joinOrder(orderId, remark)`
3.  OrderController → OrderService：`joinOrder(orderId, remark, currentUser)`
4.  OrderService → GroupOrderMember：创建参与者成员记录
5.  参与者 → 前端页面：发起支付
6.  前端页面 → OrderController：`payOrder(orderId)`
7.  OrderController → OrderService：`payOrder(orderId, currentUser)`
8.  OrderService → GroupOrderMember：更新参与者支付状态
9.  OrderService → CapitalRecord：记录支付流水

### 第四阶段：系统自动成团

1.  OrderService → GroupOrder：检查人数是否满员
2.  OrderService → GroupOrder：检查是否全员已支付
3.  OrderService → GroupOrder：更新状态为 `GROUPED`
4.  OrderService → NotificationService：发送成团通知
5.  NotificationService → 发起人 / 参与者：成团通知

------

## 4.4 图中必须体现的规则

这张图必须体现以下规则：

1.  发起人发布拼单后自动加入成员列表
2.  发起人也必须完成支付
3.  参与者加入后需要支付才真正占用名额
4.  只有当**人数满员且所有参与者都已支付**时，拼单才会进入 `GROUPED`
5.  成团后系统向所有成员发送通知

这些都是需求中的关键业务规则，画顺序图时不能漏。系统解决方案中明确采用平台代收代付、所有参与者（含发起人）支付完成后自动成团，并在成团后推送状态变更通知。

## 5. 顺序图二：上传凭证到确认收货完成

## 5.1 绘图目标

该顺序图用于描述拼单在成团之后的正常闭环过程，包括：

-   发起人上传订单凭证
-   系统根据实际金额计算差额退款
-   发起人确认送达
-   参与者确认收货
-   系统自动确认超时未响应成员
-   系统完成拼单并结算给发起人

------

## 5.2 参与对象

建议从左到右排列如下：

1.  发起人
2.  参与者
3.  前端页面
4.  OrderController
5.  OrderService
6.  OrderReceipt
7.  CapitalRecord
8.  NotificationService
9.  OrderScheduler
10.  GroupOrderMember
11.  GroupOrder

对象可以适当合并，但 `OrderScheduler` 必须出现，因为自动确认收货是定时任务触发的。

------

## 5.3 主要消息流程

### 第一阶段：上传订单凭证

1.  发起人 → 前端页面：上传订单截图并填写实际金额、预计送达时间区间
2.  前端页面 → OrderController：`uploadReceipt(orderId, receiptForm)`
3.  OrderController → OrderService：`uploadReceipt(orderId, receiptForm, currentUser)`
4.  OrderService → OrderReceipt：保存凭证记录
5.  OrderService → GroupOrder：更新实际总金额、预计送达时间区间
6.  OrderService → GroupOrder：更新状态为 `WAIT_DELIVERY`

### 第二阶段：差额退款

1.  OrderService：比较实际总金额与预估总金额
2.  若实际金额 < 预估金额：
    -   OrderService → CapitalRecord：为所有成员生成差额退款流水
    -   OrderService → GroupOrderMember：累计退款金额
3.  若实际金额 ≥ 预估金额：
    -   系统不生成差额退款流水

这里要体现：
**差额退款只在实际金额低于预估金额时发生，并且只是退差额，不是整单退款。**

### 第三阶段：确认送达

1.  发起人 → 前端页面：点击“商品已送达”
2.  前端页面 → OrderController：`markDelivered(orderId)`
3.  OrderController → OrderService：`markDelivered(orderId, currentUser)`
4.  OrderService → GroupOrder：更新状态为 `WAIT_RECEIVE`
5.  OrderService → GroupOrderMember：将所有成员收货状态更新为 `WAIT_CONFIRM`
6.  OrderService → NotificationService：发送可取货通知
7.  NotificationService → 参与者：通知取货

### 第四阶段：确认收货

1.  参与者 → 前端页面：点击确认收货
2.  前端页面 → OrderController：`confirmReceive(orderId)`
3.  OrderController → OrderService：`confirmReceive(orderId, currentUser)`
4.  OrderService → GroupOrderMember：更新成员状态为 `RECEIVED`

### 第五阶段：自动确认收货

1.  OrderScheduler → OrderService：定时检查待确认成员
2.  OrderService → GroupOrderMember：将超时成员状态更新为 `AUTO_RECEIVED`

这里必须写明：
**发起人确认送达后，参与者若 30 分钟内未点击确认收货，系统自动确认收货。**

### 第六阶段：完成拼单与结算

1.  OrderService → GroupOrder：检查是否所有成员均已收货
2.  OrderService → GroupOrder：更新状态为 `COMPLETED`
3.  OrderService → CapitalRecord：生成结算给发起人的流水
4.  OrderService → NotificationService：发送拼单完成通知

------

## 5.4 图中必须体现的规则

这张图必须体现以下规则：

1.  拼单成团后，发起人必须上传订单凭证
2.  上传凭证时要填写实际总金额和预计送达时间区间
3.  若实际金额低于预估金额，系统自动退差额
4.  发起人确认送达后，拼单转为 `WAIT_RECEIVE`
5.  参与者确认收货后更新状态
6.  超时 30 分钟未确认的成员由系统自动确认
7.  所有成员都收货后，拼单进入 `COMPLETED`
8.  拼单完成后系统生成对发起人的结算流水

这些规则都来自你们系统对交付机制和多退少补机制的设计要求。需求中明确：平台采用伪支付与伪退款；发起人上传凭证后系统按实际金额处理差额；参与者 30 分钟内未确认收货则系统默认完成收货。

------

## 6. 顺序图三：投诉提交与管理员处理

## 6.1 绘图目标

该顺序图用于描述异常流程中从投诉提交到管理员处理完成的全过程。

它要体现：

-   什么情况下会开放投诉
-   用户如何提交投诉
-   管理员如何处理投诉
-   系统如何退款、扣分并更新状态
-   系统如何通知相关用户

------

## 6.2 参与对象

建议从左到右排列如下：

1.  参与者（投诉方）
2.  管理员
3.  前端页面
4.  ComplaintController
5.  ComplaintService
6.  OrderService
7.  CapitalRecord
8.  CreditChangeRecord
9.  NotificationService
10.  Complaint
11.  GroupOrder

------

## 6.3 主要消息流程

### 第一阶段：投诉开放

1.  OrderScheduler / 系统：检测异常拼单
2.  系统 → GroupOrder：标记 `complaintOpened = true`

异常条件包括：

-   成团后 30 分钟内未上传凭证
-   超过预计最晚送达时间 60 分钟仍未确认送达

### 第二阶段：用户提交投诉

1.  参与者 → 前端页面：选择投诉类型并填写说明
2.  前端页面 → ComplaintController：`submitComplaint(complaintForm)`
3.  ComplaintController → ComplaintService：`submitComplaint(complaintForm, currentUser)`
4.  ComplaintService → Complaint：创建投诉记录
5.  ComplaintService → Complaint：设置状态为 `PENDING`
6.  ComplaintService → NotificationService：通知管理员有新投诉

### 第三阶段：管理员处理投诉

1.  管理员 → 前端页面：查看投诉详情并点击处理
2.  前端页面 → AdminController：`handleComplaint(complaintId)`
3.  AdminController → ComplaintService：`handleComplaint(complaintId, currentAdmin)`

### 第四阶段：系统执行固定处理逻辑

1.  ComplaintService → OrderService：执行投诉处理
2.  OrderService → CapitalRecord：为所有已支付成员生成退款流水
3.  OrderService → GroupOrder：更新拼单状态为 `CANCELED`
4.  ComplaintService → CreditChangeRecord：为发起人生成扣分记录
5.  ComplaintService → Complaint：更新投诉状态为 `PROCESSED`
6.  ComplaintService → Complaint：记录处理管理员

### 第五阶段：发送处理结果通知

1.  ComplaintService → NotificationService：发送投诉处理结果通知
2.  NotificationService → 参与者：投诉处理结果通知
3.  NotificationService → 被投诉人：投诉处理结果通知

------

## 6.4 图中必须体现的规则

这张图必须体现以下规则：

1.  投诉只保留两类：
    -   `NOT_PURCHASED`
    -   `FAKE_RECEIPT`
2.  管理员处理投诉时不做复杂仲裁，采用固定处理逻辑：
    -   批量退款
    -   扣除发起人信用分
    -   更新拼单状态为 `CANCELED`
    -   更新投诉状态为 `PROCESSED`
3.  投诉处理完成后系统要向投诉方和被投诉方发送通知

需求中明确指出，纠纷处理采用极简规则，管理员只需根据投诉类型和简要说明进行一键处理，无需复杂证据审核。

## 8. 推荐版式

### 8.1 顺序图推荐版式

所有顺序图统一采用：

-   左侧：参与者
-   中间：前端页面、控制类、服务类
-   右侧：实体类、支撑服务

这样可以保证：

-   用户操作在左边
-   系统处理在中间
-   数据与支撑能力在右边

### 8.2 协作图推荐版式

协作图建议采用“中心对象 + 四周对象”的布局：

#### 中心位置

-   OrderService

#### 左侧

-   发起人
-   参与者
-   前端页面

#### 上方

-   OrderController

#### 右侧

-   GroupOrder
-   GroupOrderMember
-   OrderReceipt
-   CapitalRecord

#### 下方

-   NotificationService
-   OrderScheduler

这样布局最能体现 OrderService 作为拼单主流程核心调度者的作用。

## 11. 组员绘图时的注意事项

1.  顺序图必须有生命线和激活条，不要只画箭头。
2.  协作图必须有消息编号，不要只连线不编号。
3.  三张顺序图不要使用不同命名体系。
4.  `OrderScheduler` 只应出现在涉及自动化处理的图中。
5.  `NotificationService` 在成团、送达、投诉处理等场景下要体现出来。
6.  顺序图中的消息名称尽量用方法名风格，如 `createOrder()`、`payOrder()`、`submitComplaint()`。
7.  不要把“数据库”直接画成顺序图对象，顺序图优先表现业务对象和服务对象。
8.  投诉处理图中不要画复杂审核流，因为系统没有复杂仲裁设计。
