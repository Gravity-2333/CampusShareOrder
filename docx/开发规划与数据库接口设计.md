## 2. 当前项目现状评估

### 2.1 前端现状

当前 `frontend` 已完成的主要是项目初始化骨架，包括：

- Vue 3 + Vite + Element Plus 基础环境
- 应用入口文件
- 基础样式文件
- 路由初始化
- 登录页占位页面

当前问题：

- 大部分 `views`、`api`、`stores`、`layout` 文件仍为空
- 路由目前只有 `/login`
- 还没有形成真实的页面跳转链路
- 还没有形成前后端接口调用结构

结论：

- 前端目前属于页面目录和技术栈已搭好，但业务页面尚未真正进入实现阶段

### 2.2 后端现状

当前 `backend` 已完成的主要是：

- Spring Boot 3 项目初始化
- Web 与 Spring Security 基础依赖
- 测试接口 `/api/test`
- 最小安全配置

当前问题：

- 当前后端接口基本未开始实现
- `application.yml` 为空
- 还未接入 MySQL
- 尚未接入 JWT
- 尚未接入 MyBatis-Plus
- 尚未建立实体、Mapper、Service、DTO、VO 结构
- 现有类名如 `api`、`enums`、`exception`、`impl` 不适合后续扩展

结论：

- 后端目前属于脚手架阶段，尚未进入真实业务开发阶段

### 2.3 综合判断

当前阶段最重要的工作不是继续堆页面，而是优先完成以下内容：

1. 明确并统一业务规则
2. 明确状态流转
3. 明确数据库表结构
4. 明确前后端接口契约
5. 明确开发优先级

否则后续很容易出现以下问题：

- 前端页面已经写了，但后端数据结构不匹配
- 数据库字段不够用，被迫返工
- 状态定义不一致，流程闭环做不起来
- 投诉、退款、信用分逻辑互相冲突

## 3. 开发组统一规则

以下规则作为当前开发基线：

### 3.1 统一时间规则

- 自动确认收货时间：统一为 `30 分钟`
- 成团后上传凭证超时时间：统一为 `30 分钟`
- 超过预计最晚送达时间后，额外等待 `60 分钟` 才开放投诉通道

### 3.2 统一状态流转规则

拼单主状态统一为：

1. `OPEN`：拼单中
2. `GROUPED`：已成团
3. `WAIT_DELIVERY`：待送达
4. `WAIT_RECEIVE`：待收货
5. `COMPLETED`：已完成
6. `CANCELED`：已取消

统一口径：

- 发起人发布拼单后进入 `OPEN`
- 人数满员且所有参与者支付完成后进入 `GROUPED`
- 发起人上传凭证后进入 `WAIT_DELIVERY`
- 发起人点击“商品已送达”后进入 `WAIT_RECEIVE`
- 所有人确认收货或自动确认后进入 `COMPLETED`
- 出现超时取消、管理员强制取消、投诉处理成立时进入 `CANCELED`

### 3.3 差额退款规则

统一口径：

- 若 `实际总金额 < 预估总金额`，系统自动执行差额退款
- 差额退款只退差额，不是整单退款
- 用户收到差额退款后，仍然保留在当前拼单流程中，继续等待收货
- 因此，成员状态不能因为差额退款直接变成“已退款终态”

### 3.4 暂不实现范围

以下内容暂不纳入当前课程项目核心开发范围：

- 用户互评
- 公告发布
- 真实支付对接
- 复杂证据审核
- 多角色审批流
- 复杂风控限制

### 3.5 时间区间字段规则

“预计送达时间区间”不建议只存为一个字符串，应拆成两个字段：

- `expected_delivery_start_at`
- `expected_delivery_end_at`

这样更利于：

- 定时任务判断
- 超时投诉判断
- 后续接口结构清晰

## 4. 核心业务状态设计

## 4.1 拼单主状态

### `OPEN`

说明：

- 拼单已发布，正在等待加入和支付

进入条件：

- 发起人创建拼单成功

退出条件：

- 满员且全员支付完成 -> `GROUPED`
- 截止时间到仍未成团 -> `CANCELED`

### `GROUPED`

说明：

- 拼单已成团，等待发起人上传凭证

进入条件：

- 拼单满员且所有成员支付完成

退出条件：

- 发起人上传凭证 -> `WAIT_DELIVERY`
- 超时未上传凭证，经处理取消 -> `CANCELED`

### `WAIT_DELIVERY`

说明：

- 发起人已上传凭证，等待实际送达

进入条件：

- 发起人上传凭证、填写实际金额和预计送达时间区间成功

退出条件：

- 发起人点击“商品已送达” -> `WAIT_RECEIVE`
- 超过预计最晚送达时间后超时，经投诉处理取消 -> `CANCELED`

### `WAIT_RECEIVE`

说明：

- 商品已送达，等待参与者确认收货

进入条件：

- 发起人点击“商品已送达”

退出条件：

- 全体成员确认收货或自动确认 -> `COMPLETED`

### `COMPLETED`

说明：

- 拼单流程正常闭环完成

进入条件：

- 所有人确认收货完成

### `CANCELED`

说明：

- 拼单提前终止

进入条件：

- 截止未成团
- 超时未上传凭证
- 管理员强制取消
- 投诉处理成立

## 4.2 成员状态建议

不建议只用一个字段表达成员在拼单中的所有状态，建议拆成三部分：

### 1. 加入状态 `join_status`

- `ACTIVE`：正常参与中
- `EXITED`：用户主动退出
- `CANCELED`：因拼单取消而结束

### 2. 支付状态 `pay_status`

- `UNPAID`：未支付
- `PAID`：已支付

### 3. 收货状态 `receive_status`

- `NOT_READY`：尚未进入收货阶段
- `WAIT_CONFIRM`：待确认收货
- `RECEIVED`：主动确认收货
- `AUTO_RECEIVED`：系统自动确认收货

这样拆分后更适合表达以下场景：

- 用户已支付但拼单还未成团
- 用户已收到差额退款但仍在流程中
- 用户未主动确认收货但被系统自动确认

## 5. ER 设计说明

## 5.1 实体关系概览

核心实体包括：

- 用户表 `user_account`
- 管理员表 `admin_account`
- 拼单主表 `group_order`
- 拼单成员表 `group_order_member`
- 订单凭证表 `order_receipt`
- 投诉表 `complaint`
- 资金流水表 `capital_record`
- 信用分变更表 `credit_change_record`
- 通知表 `notification`
- 操作日志表 `operation_log`

### 关系说明

- 一个用户可以发起多个拼单
- 一个用户可以参加多个拼单
- 一个拼单可以有多个成员
- 一个拼单最多有一条凭证主记录
- 一个拼单可以关联多条投诉记录
- 一个用户可以拥有多条信用分变更记录
- 一个拼单可以产生多条支付、退款、结算流水
- 一个用户可以收到多条通知
- 系统、用户、管理员的关键操作都应记录到日志

## 7. 建表 SQL 参考

```sql
CREATE TABLE user_account (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,

  credit_score INT NOT NULL DEFAULT 80,
  status VARCHAR(20) NOT NULL DEFAULT 'NORMAL',

  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP

  id BIGINT PRIMARY KEY AUTO_INCREMENT,

  deadline_at DATETIME NOT NULL,
  receipt_upload_deadline_at DATETIME,
  expected_delivery_start_at DATETIME,
  expected_delivery_end_at DATETIME,
  delivered_at DATETIME,

  related_complaint_id BIGINT,
```

## 9. 接口设计原则

### 9.1 统一前缀

所有业务接口统一使用：

- `/api`

### 9.2 统一返回结构

建议统一返回结构：

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

建议约定：

- `code = 0` 表示成功
- 非 `0` 表示失败

### 9.3 鉴权建议

建议：

- 普通用户登录使用独立 token
- 管理员登录使用独立 token
- 前端通过请求头携带 token

请求头示例：

```http
Authorization: Bearer xxxxx
```

## 10.1 认证模块

### 1. 普通用户注册

- 方法：`POST`
- 路径：`/api/auth/register`

请求参数：

```json
{
  "phone": "13800138000",
  "password": "Test@123",
  "nickname": "张三"
}
```

### 2. 普通用户登录

- 方法：`POST`
- 路径：`/api/auth/login`

请求参数：

```json
{
  "phone": "13800138000",
  "password": "Test@123"
}
```

响应示例：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "token": "user-jwt-token",
    "userInfo": {
      "id": 1,
      "nickname": "张三",
      "phone": "13800138000",
      "isVerified": true,
      "creditScore": 80
    }
  }
}
```

### 3. 管理员登录

- 方法：`POST`
- 路径：`/api/auth/admin/login`

请求参数：

```json
{
  "username": "admin",
  "password": "Admin@123"
}
```

- 方法：`GET`
- 路径：`/api/auth/me`

### 5. 退出登录

- 方法：`POST`
- 路径：`/api/auth/logout`

- 课程项目中可仅做前端删除 token

## 13. 自动任务设计建议

课程项目虽然可以简化，但以下自动逻辑仍建议统一放到任务扫描器中处理：

### 13.1 自动取消未成团拼单

规则：

- 截止时间到且未成团
- 自动取消拼单
- 自动为已支付用户执行整单退款

### 13.2 自动开放凭证超时投诉

规则：

- 拼单已成团
- 发起人 30 分钟内未上传凭证
- 系统标记投诉通道开放

### 13.3 自动开放送达超时投诉

规则：

- 已上传凭证
- 超过 `expected_delivery_end_at + 60 分钟`
- 发起人仍未点击“商品已送达”
- 系统标记投诉通道开放

### 13.4 自动确认收货

规则：

- 发起人已点击“商品已送达”
- 成员 30 分钟内未确认收货
- 系统自动确认收货

### 13.5 自动完成拼单

规则：

- 所有人都已确认收货
- 系统自动更新拼单状态为 `COMPLETED`
- 给发起人结算实际金额
