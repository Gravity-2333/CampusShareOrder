# 部署图绘制说明

## 1. 绘图目标

部署图用于展示校园拼单平台在实际运行环境中的**节点组成、软件部署位置以及节点之间的连接关系**。
该图的重点不是描述业务流程，也不是描述数据库实体关系，而是从“系统运行在哪里、各部分如何连接、前后端如何部署、数据和文件如何存储”的角度，表达整个系统的运行结构。

本图需要体现以下核心信息：

1.  用户通过浏览器访问系统；
2.  前端应用以 Web 方式部署；
3.  后端采用单体 Spring Boot 应用部署；
4.  后端连接 MySQL 数据库；
5.  后端访问本地文件存储，用于保存订单凭证截图；
6.  管理员与普通用户使用同一套系统部署结构，但访问入口角色不同；
7.  部署结构与系统总体架构设计保持一致。

部署图必须和你们的概要设计文字、总体架构图、数据库设计保持一致，不能画出实际不会部署的组件，也不能漏掉实际必须存在的运行节点。

------

## 2. 绘图风格要求

### 2.1 总体风格

部署图采用**标准 UML 部署图风格**绘制。
推荐使用以下图形元素：

-   **节点（Node）**：表示运行环境或部署节点，一般画成带立体感的矩形框，也可以画成普通矩形框但要清楚标明是节点
-   **制品（Artifact）或部署内容**：表示部署在节点内部的软件或资源
-   **连接线**：表示节点之间的通信关系
-   **连接说明文字**：在连线上标注通信方式，如 HTTP、REST API、JDBC、File I/O

### 2.2 部署图不要画成什么

部署图**不要画成**：

-   架构图
-   流程图
-   类图
-   ER 图
-   网络拓扑超复杂图

也不要把部署图画成“某层有哪些模块”的矩形堆叠图。
部署图关注的是**运行节点和连接关系**，不是逻辑分层。

------

## 3. 部署范围

根据你们当前系统设计和实现方案，部署图建议包含以下 **5 个主要节点**：

1.  客户端设备节点
2.  前端应用节点
3.  后端服务节点
4.  数据库节点
5.  文件存储节点

如果组员画图能力较强，可以额外补一个“可选缓存节点 Redis”，但建议作为虚线可选节点处理，不要画成主干节点。因为你们当前设计里 Redis 不是系统运行的硬依赖。

------

## 4. 各节点详细绘制要求

------

### 4.1 节点一：客户端设备节点

#### 节点名称

建议命名为：

**客户端设备**

#### 节点作用

表示系统使用者访问平台时所使用的终端设备。

#### 节点内部内容

内部建议放 2～3 个文字项：

-   普通用户浏览器
-   管理员浏览器
-   PC / 移动端浏览器

#### 说明

这里不需要区分安卓、iOS、桌面应用、小程序等细节，因为你们当前系统是浏览器访问模式，本质上是 Web 系统。
普通用户和管理员只是访问角色不同，不代表需要画成两套独立的客户端系统。

#### 画法建议

-   放在整张图最左侧或最上方
-   节点内部不要写过多解释，简洁即可

------

### 4.2 节点二：前端应用节点

#### 节点名称

建议命名为：

**前端应用节点**

#### 节点内部部署内容

建议在节点内部写：

-   Vue 3 Web App
-   用户端页面
-   管理端页面
-   静态资源（HTML / JS / CSS）

#### 说明

该节点表示前端系统运行环境。前端不直接连接数据库，也不直接访问文件系统，而是通过后端提供的 REST API 获取和提交数据。
管理端和用户端前端页面虽然功能不同，但本质上可以部署在同一个前端工程内，因此在部署图中不需要拆成两个前端服务器节点。

#### 画法建议

-   放在客户端设备节点之后
-   节点大小略大于客户端节点
-   节点内部使用 3～4 行文字说明即可

------

### 4.3 节点三：后端服务节点

#### 节点名称

建议命名为：

**后端服务节点**

#### 节点内部部署内容

该节点是部署图的核心，内部建议写：

-   Spring Boot Application
-   用户模块
-   拼单模块
-   信任与投诉模块
-   管理模块
-   定时任务模块

如果版面较紧，也可以简化为：

-   Spring Boot Application
-   REST API
-   Scheduler

但更推荐写出主要业务模块，这样可以和总体架构图保持一致。

#### 说明

后端采用单体应用部署方式，不拆分成多个微服务，不引入服务注册、网关集群或 RPC 通信。
后端服务节点负责：

-   身份认证与权限控制
-   拼单主业务处理
-   投诉处理
-   管理后台业务
-   状态流转
-   文件上传管理
-   定时任务自动执行

#### 画法建议

-   放在前端应用节点右侧或下方
-   是整张图中最核心的节点，尺寸可适当大一些
-   内部内容以“Spring Boot + 主要模块”组织

------

### 4.4 节点四：数据库节点

#### 节点名称

建议命名为：

**数据库节点**

#### 节点内部部署内容

建议写：

-   MySQL 8
-   campus_share_order Schema

如果组员不想写 schema 名，也可以只写：

-   MySQL 数据库

#### 说明

该节点用于存储系统业务数据和日志数据，主要包括：

-   用户数据
-   拼单数据
-   拼单成员关系数据
-   订单凭证元数据
-   投诉数据
-   信用分记录
-   资金流水
-   通知数据
-   操作日志

注意这里说的是“元数据”，即订单凭证图片本体不一定在数据库中，而是存图片路径。

#### 画法建议

-   放在后端服务节点右侧或下方
-   不需要在节点内部列出 10 张表，否则太密
-   若确实想写，最多写 3～5 个代表性表名，如：
    -   user_account
    -   group_order
    -   group_order_member
    -   complaint
    -   capital_record

------

### 4.5 节点五：文件存储节点

#### 节点名称

建议命名为：

**文件存储节点**

#### 节点内部部署内容

建议写：

-   Receipt Upload Directory
-   订单凭证图片

或者直接中文写：

-   订单凭证文件目录
-   上传图片存储

#### 说明

该节点表示系统中上传凭证截图的本地文件存储位置。
因为你们系统中“上传订单凭证”是核心功能，且文件采用本地文件系统方式保存，因此部署图中必须单独画出文件存储节点，不能默认省略。

#### 画法建议

-   放在数据库节点旁边或下方
-   与数据库节点并列，作为数据层中的另一类存储资源

------

## 5. 可选节点说明

### 5.1 Redis 节点

#### 是否必须绘制

**不是必须**。

#### 如果要画

只能画成一个**虚线框的可选节点**，名称写：

**Redis（可选）**

内部可写：

-   Token 缓存
-   临时状态缓存

#### 为什么不能画成主干节点

因为你们当前设计口径是：系统第一版可以不依赖 Redis，仅依靠 MySQL 和 Spring 定时任务也可以完成全部核心功能。如果把 Redis 画成主干部署节点，会导致部署图和最终实现可能不一致。

------

## 6. 节点之间的连接关系

这一部分是部署图最关键的逻辑，负责画图的组员必须严格按下面的关系连接。

------

### 6.1 客户端设备节点 → 前端应用节点

#### 连接说明

客户端通过浏览器访问前端应用。

#### 连线标注建议

在线上标注：

-   HTTP / HTTPS

#### 说明

这条连接表示普通用户和管理员通过浏览器打开前端页面，访问系统入口。

------

### 6.2 前端应用节点 → 后端服务节点

#### 连接说明

前端通过接口调用后端。

#### 连线标注建议

在线上标注：

-   REST API
-   HTTP / JSON

#### 说明

前端不直接访问数据库，而是统一通过 REST 接口与后端通信。
这条连接必须明确画出，因为它体现了前后端分离结构。

------

### 6.3 后端服务节点 → 数据库节点

#### 连接说明

后端服务访问 MySQL 数据库进行数据读写。

#### 连线标注建议

在线上标注：

-   JDBC
-   SQL

#### 说明

这条连接体现后端与数据库之间的持久化访问关系。

------

### 6.4 后端服务节点 → 文件存储节点

#### 连接说明

后端服务将上传的凭证图片保存到本地文件目录，并根据业务需要读取文件路径。

#### 连线标注建议

在线上标注：

-   File I/O
-   文件读写

#### 说明

这条连接必须出现，因为订单凭证上传属于你们系统的关键功能，如果部署图里没有文件存储关系，会导致部署结构不完整。

------

### 6.5 后端服务节点 ↔ Redis 节点（如绘制）

#### 连接说明

后端服务可选使用 Redis 进行缓存。

#### 连线标注建议

在线上标注：

-   Cache
-   可选连接

#### 说明

如果画 Redis，这条线建议使用虚线，表示不是系统的主干必需链路。

------

## 7. 推荐版式

为了使整张图清晰美观，推荐采用**从左到右部署关系图**的形式。

### 推荐布局一：横向布局

从左到右依次摆放：

1.  客户端设备节点
2.  前端应用节点
3.  后端服务节点
4.  数据库节点
5.  文件存储节点

Redis（如画）可放在后端服务节点下方，用虚线连接。

这种布局的优点是：

-   逻辑清晰
-   连线少
-   易于阅读
-   和“访问 → 服务 → 数据”的思路一致

------

### 推荐布局二：分层布局

从上到下分三层：

#### 第一层

-   客户端设备节点

#### 第二层

-   前端应用节点
-   后端服务节点

#### 第三层

-   数据库节点
-   文件存储节点

这种布局也可以，但要求节点位置对称，否则容易显得杂乱。

------

## 8. 图中命名统一要求

为了保证与概要设计正文、架构图和后续实现一致，部署图中所有名称必须统一。

### 8.1 节点名称统一为

-   客户端设备
-   前端应用节点
-   后端服务节点
-   数据库节点
-   文件存储节点
-   Redis（可选）

### 8.2 内部部署内容统一为

-   Vue 3 Web App
-   Spring Boot Application
-   MySQL 8
-   订单凭证图片 / 文件目录

### 8.3 模块名称统一为

-   用户模块
-   拼单模块
-   信任与投诉模块
-   管理模块
-   定时任务模块

不要一会儿写“业务服务A”，一会儿写“订单中心”，一会儿又写“支付服务”。
部署图中的模块命名必须和总体架构图、概要设计正文保持一致。

------

## 9. 图中不要出现的内容

组员绘图时，部署图中不要出现以下内容：

1.  **微服务相关组件**
    -   API Gateway
    -   注册中心
    -   服务发现
    -   Nacos
    -   Dubbo
    -   Kafka
    -   RabbitMQ
2.  **与当前设计无关的复杂基础设施**
    -   CDN
    -   负载均衡集群
    -   Docker / Kubernetes
    -   对象存储 OSS / COS（除非你们最终真的要这么实现）
3.  **具体业务流程描述**
    -   如“用户点击支付”
    -   如“管理员处理投诉”
    -   这些属于顺序图或活动图内容，不属于部署图
4.  **数据库表完整字段**
    -   部署图不是数据库设计图，不要把表结构塞进去

------

## 10. 图名与图下注释建议

### 图名建议

**图 X-X 校园拼单平台系统部署图**

### 图下注释建议

本图展示校园拼单平台的运行部署结构。普通用户和管理员通过浏览器访问前端应用，前端应用通过 REST API 调用单体部署的 Spring Boot 后端服务。后端服务统一处理用户、拼单、投诉和管理等业务，并通过 MySQL 数据库存储业务数据与日志数据，通过本地文件存储保存订单凭证截图。该部署方案适用于课程项目环境，结构简洁，便于开发、测试和演示。

## 11. 与其他图的对应关系说明

为了避免不同 UML 图之间逻辑不匹配，部署图必须和其他设计图保持以下对应关系：

### 11.1 与总体架构图的对应关系

-   总体架构图中的“访问层”对应部署图中的“客户端设备”
-   总体架构图中的“表示与接口层”对应部署图中的“前端应用节点”和“后端服务节点”
-   总体架构图中的“数据层”对应部署图中的“数据库节点”和“文件存储节点”

### 11.2 与 ER 图的对应关系

-   ER 图描述数据库内部有哪些核心数据实体
-   部署图只需体现这些实体最终存放在“MySQL 数据库节点”中
-   不需要在部署图中展开实体关系

### 11.3 与后续程序实现的对应关系

-   前端最终应作为一个 Vue 工程存在
-   后端最终应作为一个单体 Spring Boot 工程存在
-   数据库最终应使用 MySQL
-   凭证图片最终应存放在本地文件目录
    因此部署图必须保持这种实现口径，不能画成未来才可能使用的复杂云端部署方式。

------

## 12. 组员绘图时的注意事项

1.  **部署图一定要体现“文件存储节点”**
    因为你们系统的订单凭证上传是正式功能，且不存数据库大字段。
2.  **不要把管理员画成单独一套系统**
    管理员只是访问角色不同，不是另一套独立部署架构。
3.  **不要把前端和后端画反**
    浏览器访问前端，前端访问后端，后端访问数据库，这个顺序必须清楚。
4.  **不要用过多花哨图标**
    重点是结构清晰，不是做海报。
5.  **文字不要写太多**
    每个节点内部列出关键部署内容即可，不要写长句子。
6.  **连线一定要有语义**
    -   浏览器到前端：HTTP/HTTPS
    -   前端到后端：REST API
    -   后端到数据库：JDBC
    -   后端到文件存储：File I/O
7.  **如要加 Redis，必须标明“可选”**
    避免老师认为你们的最终程序依赖缓存才能运行。

------

3.  节点之间连接关系明确，并标注通信方式；
4.  图中文字命名统一；
5.  图与概要设计正文、总体架构图、数据库设计保持一致。
