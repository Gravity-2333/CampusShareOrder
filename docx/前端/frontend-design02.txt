## 2. 路由守卫真值表

页面分组固定如下：

* 用户公开页：`/login` `/register`
* 用户认证页：`/verify-student`
* 用户业务页：`/orders` `/orders/:orderId` `/orders/create` `/my/orders` `/profile` `/credit` `/complaints/create` `/complaints/my`
* 管理员公开页：`/admin/login`
* 管理员业务页：`/admin/users` `/admin/orders` `/admin/complaints` `/admin/logs`

| 身份               | 用户公开页              | 用户认证页 | 用户业务页 | 管理员公开页             | 管理员业务页 | 跳转目标                                                                                  | 提示文案                  |
| ---------------- | ------------------ | ----- | ----- | ------------------ | ------ | ------------------------------------------------------------------------------------- | --------------------- |
| 未登录              | 允许                 | 拒绝    | 拒绝    | 允许                 | 拒绝     | 访问用户侧受限页时跳 `/login`；访问管理侧受限页时跳 `/admin/login`                                         | `请先登录` / `请先以管理员身份登录` |
| 普通用户（已实名、NORMAL） | 拒绝                 | 拒绝    | 允许    | 允许但提交登录会失败         | 拒绝     | 访问 `/login` `/register` 跳 `/orders`；访问 `/verify-student` 跳 `/orders`；访问管理页跳 `/orders` | `已登录` / `无管理员权限`      |
| 普通用户（未实名、NORMAL） | 拒绝                 | 允许    | 拒绝    | 允许但提交登录会失败         | 拒绝     | 登录后统一落 `/verify-student`；访问用户业务页也跳 `/verify-student`                                  | `请先完成实名认证`            |
| 管理员（NORMAL）      | 拒绝                 | 拒绝    | 拒绝    | 拒绝                 | 允许     | 访问用户侧任何页跳 `/admin/users`；访问 `/admin/login` 也跳 `/admin/users`                          | `管理员账号不进入用户端` / `已登录` |
| 被封禁用户（BANNED）    | 若本地已有 token，进入前即拦截 | 拦截    | 拦截    | 若本地已有 token，进入前即拦截 | 拦截     | 清空 token；若 `role=USER` 跳 `/login`；若 `role=ADMIN` 跳 `/admin/login`                     | `账号已被封禁，请联系管理员`       |

路由守卫执行顺序固定：

1. 先判 token 是否存在
2. 再调 `GET /api/auth/me` 恢复身份
3. 再判 `status`
4. 再判 `role`
5. 最后判 `isVerified`

## 3. 页面级接口契约

下列 JSON **全部只写 `data` 结构**。真实响应外层统一为：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

分页 `data` 统一为：

```json
{
  "list": [],
  "page": 1,
  "pageSize": 10,
  "total": 0,
  "pages": 0
}
```

系统设计文档已固定统一返回体、认证模块、用户模块、拼单模块、投诉模块、管理模块的接口边界；开发规划文档又补齐了用户页与管理页的页面映射。下面直接按最终版写死。

**body**

| 字段       | 类型     | 必填 |
| -------- | ------ | -: |

## 5. 状态与按钮矩阵

前置固定规则：

1. 所有详情页动作成功后，前端**必须**先重拉一次 `GET /api/orders/{orderId}`
2. 若当前页面来自 `/my/orders`，成功后再额外重拉一次 `GET /api/users/my-orders`
3. `提交投诉` 成功后，额外重拉一次 `GET /api/complaints/my`
4. 发起人**不显示**“确认收货”按钮；发起人点击“商品已送达”时，自己的 `receive_status` 直接视为 `RECEIVED`

| 当前用户身份  | 订单主状态                                    | pay_status | receive_status  | 显示按钮              | 禁用条件                               | 接口                                                                   | 成功后刷新                                                  |
| ------- | ---------------------------------------- | ---------- | --------------- | ----------------- | ---------------------------------- | -------------------------------------------------------------------- | ------------------------------------------------------ |
| 非成员普通用户 | `OPEN`                                   | -          | -               | `加入拼单`            | 满员、已过截止时间、已加入、本人是发起人               | `POST /api/orders/{orderId}/join`                                    | `GET /api/orders/{orderId}`                            |
| 已加入成员   | `OPEN`                                   | `UNPAID`   | `NOT_READY`     | `支付预估金额`、`退出拼单`   | `canPay=false` / `canExit=false`   | `POST /api/orders/{orderId}/pay` / `POST /api/orders/{orderId}/exit` | `GET /api/orders/{orderId}`                            |
| 已加入成员   | `OPEN`                                   | `PAID`     | `NOT_READY`     | `退出拼单`            | `canExit=false`                    | `POST /api/orders/{orderId}/exit`                                    | `GET /api/orders/{orderId}`                            |
| 发起人     | `OPEN`                                   | `UNPAID`   | `NOT_READY`     | `支付预估金额`          | `canPay=false`                     | `POST /api/orders/{orderId}/pay`                                     | `GET /api/orders/{orderId}`                            |
| 发起人     | `OPEN`                                   | `PAID`     | `NOT_READY`     | 无操作按钮，仅显示“等待满员”   | -                                  | -                                                                    | -                                                      |
| 发起人     | `GROUPED`                                | `PAID`     | `NOT_READY`     | `上传订单凭证`          | 已有凭证、`canUploadReceipt=false`      | `POST /api/orders/{orderId}/upload-receipt`                          | `GET /api/orders/{orderId}`                            |
| 普通参与成员  | `GROUPED`                                | `PAID`     | `NOT_READY`     | 无操作按钮             | -                                  | -                                                                    | -                                                      |
| 普通参与成员  | `GROUPED` 且 `complaintOpened=true`       | `PAID`     | `NOT_READY`     | `提交投诉`            | 已存在本人投诉、`canCreateComplaint=false` | `POST /api/complaints`                                               | `GET /api/orders/{orderId}` + `GET /api/complaints/my` |
| 普通参与成员  | `GROUPED` 且已投诉                           | `PAID`     | `NOT_READY`     | `已提交投诉`（禁用）       | 固定禁用                               | -                                                                    | -                                                      |
| 发起人     | `WAIT_DELIVERY`                          | `PAID`     | `RECEIVED`      | `商品已送达`           | `canMarkDelivered=false`           | `POST /api/orders/{orderId}/mark-delivered`                          | `GET /api/orders/{orderId}`                            |
| 普通参与成员  | `WAIT_DELIVERY`                          | `PAID`     | `NOT_READY`     | 无操作按钮             | -                                  | -                                                                    | -                                                      |
| 普通参与成员  | `WAIT_DELIVERY` 且 `complaintOpened=true` | `PAID`     | `NOT_READY`     | `提交投诉`            | 已存在本人投诉、`canCreateComplaint=false` | `POST /api/complaints`                                               | `GET /api/orders/{orderId}` + `GET /api/complaints/my` |
| 普通参与成员  | `WAIT_RECEIVE`                           | `PAID`     | `WAIT_CONFIRM`  | `确认收货`            | `canConfirmReceived=false`         | `POST /api/orders/{orderId}/confirm-received`                        | `GET /api/orders/{orderId}`                            |
| 普通参与成员  | `WAIT_RECEIVE`                           | `PAID`     | `RECEIVED`      | 无按钮，显示“已确认收货”     | -                                  | -                                                                    | -                                                      |
| 普通参与成员  | `WAIT_RECEIVE`                           | `PAID`     | `AUTO_RECEIVED` | 无按钮，显示“系统已自动确认”   | -                                  | -                                                                    | -                                                      |
| 发起人     | `WAIT_RECEIVE`                           | `PAID`     | `RECEIVED`      | 无操作按钮，仅显示“等待成员确认” | -                                  | -                                                                    | -                                                      |
| 任意已登录用户 | `COMPLETED`                              | 任意         | 任意              | 无操作按钮，可看凭证        | -                                  | -                                                                    | -                                                      |
| 任意已登录用户 | `CANCELED`                               | 任意         | 任意              | 无操作按钮，可看取消原因      | -                                  | -                                                                    | -                                                      |

额外固定显示规则：

| 条件                                       | 显示             |
| ---------------------------------------- | -------------- |
| `receiptInfo.hasReceipt=true`            | 显示 `查看凭证`      |
| `actionFlags.canViewReceipt=false`       | 不显示 `查看凭证`     |
| `complaintInfo.myComplaintId != null`    | 详情页不再显示 `提交投诉` |
| `currentUserMember.joinStatus != ACTIVE` | 所有成员动作按钮一律不显示  |

## 6. 表单校验规则

密码规则来自需求文档；实名认证要求绑定学号；发起拼单字段要求来自需求文档的 P0 拼单填写项。

| 字段                                                    | 最终规则                                  | 错误提示文案                          |
| ----------------------------------------------------- | ------------------------------------- | ------------------------------- |
| `phone`                                               | 必填；去首尾空格；必须匹配 `^1[3-9]\d{9}$`         | `请输入正确的11位手机号`                  |
| `password`                                            | 必填；长度 `6-20`；不得含空格；字母/数字/特殊符号至少包含 2 种 | `密码需为6-20位，且至少包含字母、数字、特殊符号中的2种` |
| `nickname`                                            | 必填；去首尾空格；长度 `2-20`                    | `昵称长度需为2-20个字符`                 |
| `studentNo`                                           | 必填；必须是 `10` 位数字                       | `学号必须为10位数字`                    |
| `productName`                                         | 必填；去首尾空格；长度 `2-30`                    | `商品名称长度需为2-30个字符`               |
| `productDesc`                                         | 选填；去首尾空格；最长 `200`                     | `商品描述不能超过200个字符`                |
| `memberCount`（对应 `totalMemberCount`）                  | 必填；整数；范围 `2-20`                       | `总人数必须为2-20之间的整数`               |
| `amount`（对应 `estimatedTotalAmount/actualTotalAmount`） | 必填；大于 `0`；最多两位小数；范围 `(0, 9999.99]`    | `金额必须大于0，且最多保留2位小数`             |
| `deadlineAt`                                          | 必填；合法时间字符串；必须晚于当前时间                   | `截止时间必须晚于当前时间`                  |
| `pickupPoint`                                         | 必填；去首尾空格；长度 `2-50`                    | `取货点长度需为2-50个字符`                |
| `remark`                                              | 选填；去首尾空格；最长 `100`                     | `备注不能超过100个字符`                  |
| `complaintContent`（对应 `content`）                      | 必填；去首尾空格；长度 `5-200`                   | `投诉说明长度需为5-200个字符`              |

前端校验执行顺序固定：

1. `trim()`
2. 必填校验
3. 格式校验
4. 长度/范围校验
5. 再发请求

## 7. 错误码与前端处理规则

|    code | 含义           | 前端处理                                          |
| ------: | ------------ | --------------------------------------------- |
|     `0` | 成功           | 正常处理                                          |
| `40001` | 参数不合法        | `ElMessage.error(message)`，停留当前页              |
| `40002` | 手机号格式错误      | 表单报错，不跳转                                      |
| `40003` | 密码格式错误       | 表单报错，不跳转                                      |
| `40004` | 学号格式错误       | 表单报错，不跳转                                      |
| `40005` | 金额格式错误       | 表单报错，不跳转                                      |
| `40006` | 截止时间不合法      | 表单报错，不跳转                                      |
| `40101` | 未登录          | 清 token，跳 `/login` 或 `/admin/login`，提示 `请先登录` |
| `40102` | token 失效/过期  | 清 token，跳登录页，提示 `登录已过期，请重新登录`                 |
| `40301` | 无权限          | 保持当前页，提示 `无权限访问`                              |
| `40302` | 账号已封禁        | 清 token，按角色跳登录页，提示 `账号已被封禁，请联系管理员`            |
| `40303` | 未实名认证        | 跳 `/verify-student`，提示 `请先完成实名认证`             |
| `40401` | 拼单不存在        | 若当前在详情页，跳 `/orders`，提示 `拼单不存在或已删除`            |
| `40402` | 投诉不存在        | 若当前在投诉详情抽屉，关闭抽屉并提示 `投诉不存在`                    |
| `40403` | 用户不存在        | 提示 `用户不存在`                                    |
| `40901` | 手机号已注册       | 注册页提示 `手机号已注册`                                |
| `40902` | 学号已绑定        | 实名页提示 `该学号已绑定其他账号`                            |
| `40903` | 当前订单状态不允许该操作 | 提示后立刻重拉 `GET /api/orders/{orderId}`           |
| `40904` | 订单已满员        | 提示后重拉详情和当前列表页                                 |
| `40905` | 已加入该拼单       | 提示后重拉详情                                       |
| `40906` | 不是该拼单成员      | 提示后重拉详情                                       |
| `40907` | 已支付，无需重复支付   | 提示后重拉详情                                       |
| `40908` | 当前阶段不允许退出    | 提示后重拉详情                                       |
| `40909` | 凭证已上传        | 提示后重拉详情                                       |
| `40910` | 当前阶段不允许确认送达  | 提示后重拉详情                                       |
| `40911` | 已确认收货，无需重复操作 | 提示后重拉详情                                       |
| `40912` | 当前未开放投诉      | 提示后重拉详情                                       |
| `40913` | 重复投诉         | 提示后重拉详情和我的投诉列表                                |
| `40914` | 发起人不能投诉自己    | 提示 `发起人不能对本单发起投诉`                             |
| `50000` | 系统异常         | `ElMessage.error('系统异常，请稍后再试')`，不跳转           |

错误处理统一执行规则：

1. `40101/40102/40302` 必须清空 Pinia 登录态
2. `40903-40913` 必须重拉当前详情数据
3. 表单类错误只做前端提示，不自动跳页
4. 详情类资源不存在时才跳回列表页

## 8. 冲突项拍板

| 冲突项     | 最终版                                                                     |
| ------- | ----------------------------------------------------------------------- |
| 我的投诉接口  | **只保留** `GET /api/complaints/my`                                        |

## 10. 开发顺序

开发规划文档已经明确，当前核心不是继续铺页面，而是先把认证、用户、拼单主流程接口和前端闭环打通；迭代一计划也把“前端请求封装、登录认证、列表、详情、创建、加入/支付、状态流转”放在第一优先级。下面直接给最小并行顺序。 

| 步骤 | 前端                                                                                        | 后端                                                                                                                                 | 完成标志                                                     |
| -- | ----------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------- |
| 1  | 安装 `axios`；落 `api/request.js`；建 `userStore`；建路由守卫；实现 `BlankLayout/UserLayout/AdminLayout` | 补 `MyBatis-Plus`、`JWT`、`Validation`、`MySQL` 依赖；建统一返回体、统一错误码、统一枚举；保留 `permitAll` 直到登录接口落地                                           | 前端可发真实请求；后端可返回统一 JSON                                    |
| 2  | 完成 `/login` `/register` `/verify-student` 页面                                              | 完成 `user_account` 表；实现 `POST /api/auth/register` `POST /api/auth/login` `GET /api/auth/me` `POST /api/users/verify-student`        | 用户能注册、登录、刷新后恢复登录态、完成实名认证                                 |
| 3  | 完成 `/orders` `/orders/create` `/orders/:orderId` 页面静态渲染                                   | 完成 `group_order` `group_order_member` 表；实现 `POST /api/orders` `GET /api/orders` `GET /api/orders/{id}`                             | 可以创建拼单、看到列表、打开详情                                         |
| 4  | 完成详情页 `加入拼单`、`支付预估金额`、`退出拼单` 按钮与刷新逻辑                                                      | 完成 `capital_record` 表；实现 `POST /api/orders/{id}/join` `POST /api/orders/{id}/pay` `POST /api/orders/{id}/exit`；在 `pay` 成功后同步判断是否成团 | 能跑通“发起人创建并支付 -> 参与者加入并支付 -> 自动成团”                        |
| 5  | 完成 `/my/orders` 页面；详情页成功动作后统一重拉数据                                                         | 实现 `GET /api/users/my-orders`                                                                                                      | 我的拼单能看到自己发起和参与的数据                                        |
| 6  | 联调完整 Happy Path                                                                           | 完成 `OPEN -> GROUPED` 同步状态流转；未成团退出退款；必要日志落库                                                                                         | `登录 -> 实名 -> 创建 -> 发起人支付 -> 参与者加入 -> 参与者支付 -> 成团` 全链路可演示 |
| 7  | 再补 `/credit` `/complaints/my`                                                             | 再补 `credit_change_record` `complaint`；实现信用分和投诉接口                                                                                   | 信任闭环可演示                                                  |
| 8  | 最后补管理员页面 `/admin/users` `/admin/orders` `/admin/complaints` `/admin/logs`                 | 最后补管理员接口、封禁/解封、强制取消、投诉处理、日志查询                                                                                                      | 管理后台可演示                                                  |
