# CampusShareOrder 前端设计方案（开发组版本）

## 1. 设计目标

本方案用于冻结校园拼单平台前端的页面职责、模块边界、路由结构、状态模型、接口映射与开发顺序，作为后续使用 Codex 编码时的统一前端输入。

前端目标不是先“把所有页面都铺出来”，而是：

1. 先建立稳定的页面骨架与路由边界；
2. 先打通核心闭环：登录 -> 列表 -> 详情 -> 发起拼单 -> 加入/支付 -> 上传凭证 -> 送达 -> 收货 -> 投诉；
3. 保证用户端与管理端共享一套前端工程，但使用不同布局和权限控制；
4. 所有状态名、接口路径、页面动作都与后端统一口径保持一致。

因此，接下来的重点不是再建更多空页面，而是把“前端架构规则”先定死。

## 3. 前端总体架构原则

### 3.1 技术栈

固定使用：

- Vue 3
- Vite
- Element Plus
- Vue Router
- Pinia
- Axios

### 3.2 架构原则

1. **一个前端工程，双端入口逻辑**
   - 用户端和管理端共用同一个 Vue 项目；
   - 通过路由前缀、布局组件和权限守卫区分。

2. **页面容器 + 业务组件分离**
   - `views/` 放页面容器；
   - `components/` 放复用业务组件；
   - 页面不堆过多重复 UI 片段。

3. **API 文件按业务域拆分**
   - `auth.js`
   - `user.js`
   - `order.js`
   - `complaint.js`
   - `admin.js`

4. **Pinia 只放全局状态**
   - 登录状态
   - 当前用户信息
   - 当前角色
   - 布局/菜单等全局 UI 状态
   - 不建议把所有列表数据都塞进 Pinia，列表和详情数据优先走页面级状态。

5. **所有状态枚举前后端统一**
   - 不允许前端自己造中文自由文本状态；
   - 前端显示中文标签时，由 `utils/enum.js` 做映射。

6. **先做闭环，再做美化**
   - 先把功能和状态流转打通；
   - UI 保持简洁、稳定、答辩友好。

## 5.3 Order 模块

这是前端核心模块。

### `OrderListView.vue`

职责：

- 展示拼单列表；
- 支持状态筛选、关键词搜索、取货点筛选、排序；

每张卡片建议显示：

- 预估人均金额
- 取货点
- 截止时间

### `OrderDetailView.vue`

职责：

- 展示完整拼单详情；
- 展示状态、成员、支付状态、时间线、投诉开放情况；
- 根据“当前用户身份 + 当前拼单状态”显示不同操作按钮。

建议拆成 5 个区块：

1. 拼单基础信息
2. 状态信息与时间节点
3. 成员与支付状态
4. 凭证 / 送达 / 收货信息
5. 操作区

- 未加入且可加入：显示“加入拼单”

- 发起人且 `GROUPED`：显示“上传订单凭证”

- 参与者且 `WAIT_RECEIVE`：显示“确认收货”
- 可投诉时：显示“发起投诉”

职责：

## 6. 路由设计

建议路由结构如下：

```js
/
/login
/register
/admin/login

/orders
/orders/create
/orders/:orderId
/my/orders
/verify-student
/profile
/credit
/complaints/create
/complaints/my
/complaints/:complaintId

/admin
/admin/dashboard
/admin/users
/admin/orders
/admin/complaints
/admin/logs
```

### 6.1 路由守卫规则

建议在 `router/index.js` 中使用 `meta`：

- `requiresAuth: true`
- `role: 'user' | 'admin'`
- `requiresVerified: true`

控制逻辑：

1. 未登录不能访问业务页；
2. 非管理员不能进入 `/admin/*`；
3. 未实名认证用户不能发起拼单、加入拼单；
4. 被封禁用户应强制退出并提示联系管理员。

## 8. 接口映射设计

## 8.1 统一返回格式

前端统一按以下格式处理：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

规则：

- `code === 0` 视为成功；
- 其他状态统一走错误提示；
- 401/403 走登录失效处理；
- 业务错误优先显示后端 `message`。

## 8.2 `api/auth.js`

## 9. 请求层设计

`request.js` 需要统一处理：

1. `baseURL`
2. 请求超时
3. token 注入
4. 响应拦截
5. 统一错误提示
6. 登录失效处理

建议 token 存储：

- 普通用户：`USER_TOKEN`

这样做的原因：

- 用户端和管理端账号体系分离；
- 避免角色混用；
- 路由守卫更清晰。

## 11. 页面交互规则

### 11.1 创建拼单

- 未实名认证不能创建；
- 表单提交成功后跳详情；
- 详情页提示发起人需完成支付。

### 11.2 加入拼单

- 仅 `OPEN` 状态可加入；
- 满员或已取消不可加入；
- 加入时允许填写备注。

### 11.3 支付

- 使用伪支付接口；
- 支付成功后刷新详情、成员状态、时间线。

### 11.4 上传凭证

- 仅发起人可操作；
- 仅 `GROUPED` 状态可上传；
- 需提交图片地址、实际金额、预计送达时间区间。

### 11.5 确认送达

- 仅发起人可操作；
- 仅 `WAIT_DELIVERY` 状态可操作。

### 11.6 确认收货

- 仅参与者可操作；
- 仅 `WAIT_RECEIVE` 状态可操作；
- 发起人本人不应显示“确认收货”按钮。

### 11.7 投诉

- 仅可在投诉通道开放后提交；
- 前端不做复杂证据审核；
- 投诉提交成功后跳转我的投诉页。

---

## 12. 通知设计（轻量版）

虽然当前前端目录中没有单独的通知页面，但系统设计中通知是正式支撑模块。

第一版前端建议：

- 不单独做通知中心页面；
- 在关键动作后使用消息提示 + 刷新数据；
- 在 `UserLayout` 顶部预留通知徽标位；
- 后续如时间允许，再补一个下拉通知面板。

这样能保证与系统设计一致，同时不增加过多页面负担。

## 13. 命名与规范建议

### 13.1 统一修正项

建议尽快统一：

- `AdminDashBoardView.vue` -> `AdminDashboardView.vue`
- `public/轮播_256_50rd.ico` -> `public/favicon.ico`

### 13.2 命名规则

- 页面组件：`XxxView.vue`
- 布局组件：`XxxLayout.vue`
- 复用业务组件：`XxxCard.vue` / `XxxTable.vue` / `XxxDialog.vue`
- API 方法：动词开头，如 `getOrderDetail`
- Store 方法：`set / fetch / clear / update` 风格

### 13.3 页面不要做的事情

- 不要在页面里直接写 axios；
- 不要在页面里散写状态文案；
- 不要把权限判断硬编码成很多 if-else；
- 不要把所有表单都塞到一个巨型页面里。

## 15. 给 Codex 的执行约束

后续把本文件喂给 Codex 时，建议同时附上这几条硬约束：

1. 严格沿用当前目录结构，不要擅自改成别的框架风格；
2. 保持 Vue 3 `script setup` 风格；
3. UI 组件统一使用 Element Plus；
4. 路由按用户端 / 管理端分组；
5. 所有页面先完成“能跑 + 能请求 + 能展示 + 能交互”的版本；
6. 不实现真实支付、不引入 WebSocket、不引入复杂状态机库；
7. 所有状态枚举统一从 `utils/enum.js` 获取；
8. 所有请求统一从 `api/*` 进入；
9. 所有需要鉴权的页面统一走路由守卫；
10. 先实现用户端闭环，再补管理端。

## 16. 一句话结论

你们当前前端已经有“可用于课程项目开发的目录骨架”，现在最需要的不是继续建空文件，而是按本方案把：

- 页面职责
- 路由边界
- 状态模型
- 接口映射
- 权限规则

一次性定下来。这样后续无论是你们手写，还是交给 Codex 生成，都会稳定得多。
