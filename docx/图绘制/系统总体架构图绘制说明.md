# 系统总体架构图绘制说明

## 1. 绘图目标

系统总体架构图用于展示校园拼单平台从用户访问到业务处理，再到数据存储的整体逻辑结构。该图不用于描述具体业务流程，不用于描述数据库表关系，也不用于描述类之间的调用细节，而是从**系统分层与模块协作**的角度，展示整个平台的总体技术结构。

本图需要表达以下核心思想：

1.  系统采用**前后端分离的 B/S 架构**；
2.  用户通过浏览器访问前端页面；
3.  前端通过 REST API 调用后端服务；
4.  后端采用单体应用的分层结构；
5.  业务层按模块划分为用户模块、拼单模块、信任与投诉模块、管理模块；
6.  公共支撑能力包括认证鉴权、定时任务、通知服务和日志服务；
7.  数据层由 MySQL 数据库和本地文件存储组成；
8.  架构图要能与后续部署图、数据库设计、类图保持一致。

------

## 2. 绘图风格要求

### 2.1 总体风格

本图严格参考老师给出的“架构图”示例风格进行绘制，采用**自上而下分层矩形框图**的形式，而不是流程图、泳道图或 UML 类图形式。

整体要求如下：

-   采用**纵向分层结构**
-   每一层用一个较大的外框表示
-   每层内部再放置功能模块小框
-   层与层之间上下排列，用箭头或线条表示调用方向
-   尽量保持整齐对称，避免自由散点式布局
-   整体视觉上应接近老师示例：上层访问终端，中间业务与支撑模块，底层数据库与存储

### 2.2 图中不要出现的内容

架构图中**不要出现**以下内容：

-   具体接口路径，如 `/api/orders`
-   具体数据库字段名
-   具体类名，如 `OrderServiceImpl`
-   具体时序过程
-   具体状态值，如 `OPEN`、`GROUPED`
-   太细的业务步骤，如“用户点击确认收货”
-   微服务、MQ、RPC、网关集群等你们项目当前并未采用的复杂结构

因为你们当前采用的是**课程项目单体架构**，不是微服务架构。如果画出 API Gateway、MQ、RPC、注册中心等内容，会和后续程序实现不一致。

------

## 3. 架构图的总体分层

系统总体架构图建议划分为 **4 层**，从上到下依次为：

1.  访问层
2.  表示与接口层
3.  业务服务层
4.  数据层

整个图应呈现为“上层访问、中层处理、下层存储”的结构。

------

## 4. 各层详细绘制要求

------

### 4.1 第一层：访问层

#### 层名称

建议命名为：

**访问层**

#### 位置

位于整张图最上方，占一整行。

#### 层内模块

访问层内部放置 3 个并列的小框，分别表示三类访问终端：

1.  **PC 浏览器**
2.  **移动端浏览器**
3.  **管理员浏览器**

#### 说明

这里不要画成“APP”“小程序”“其他终端”等，因为你们目前需求和实现方案里本质上都是浏览器访问，只是角色不同。普通用户和管理员都通过 Web 页面访问系统，只不过访问的页面和权限不同。

#### 画法建议

-   三个小框横向排列
-   每个框中可配一个简单图标，也可以只写文字
-   访问层外部再套一个大框，标明“访问层”

------

### 4.2 第二层：表示与接口层

#### 层名称

建议命名为：

**表示与接口层**

也可以拆成两行子结构：

-   前端表示层
-   后端接口层

如果版面允许，建议拆分，更清楚。

#### 推荐内部结构

这一层内部建议画成左右或上下两个部分：

#### （1）前端表示层

放一个大框，名称写：

**Vue 前端应用**

可以在框内用较小字号列出：

-   用户端页面
-   管理端页面
-   路由管理
-   状态管理
-   请求封装

这里不是要求把所有页面名都写进去，而是突出前端承担的职责：

-   页面展示
-   用户交互
-   状态管理
-   调用后端接口

#### （2）后端接口层

在前端表示层之下或右侧，放一个大框，名称写：

**REST API 接口层**

框内建议写两类接口：

-   用户端接口
-   管理端接口

#### 说明

这一层的核心是表达：

-   前端不是直接操作数据库
-   前端必须通过 REST API 调用后端
-   管理端也是通过同一后端接口体系完成业务处理

#### 连接关系

-   访问层三个浏览器都连到“Vue 前端应用”
-   “Vue 前端应用”再连到“REST API 接口层”

------

### 4.3 第三层：业务服务层

这一层是整张图的核心，需要画得最完整。

#### 层名称

建议命名为：

**业务服务层**

#### 总体结构

这一层建议划分为两部分：

1.  左侧或上方：**公共支撑模块**
2.  中间主体：**核心业务模块**

如果版面够宽，也可以右侧再单独列一个“系统管理支撑”栏，但不是必须。

------

#### 4.3.1 公共支撑模块

建议在业务服务层内单独画一个较窄的区域，命名为：

**公共支撑模块**

内部放 4 个小框：

1.  **认证鉴权**
    -   对应 JWT、Spring Security
    -   负责普通用户和管理员身份认证、权限控制
2.  **定时任务**
    -   负责超时取消、开放投诉、自动确认收货、自动完成拼单
3.  **通知服务**
    -   负责成团通知、退款通知、投诉处理结果通知、可取货通知
4.  **日志服务**
    -   负责操作日志、业务日志、异常记录

#### 说明

这里体现的是“支撑性能力”，不是业务模块本身，但它们会被多个业务模块共同使用。

------

#### 4.3.2 核心业务模块

在业务服务层中央画一个较大的主体框，名称写：

**核心业务模块**

内部放置 4 个并列或 2×2 排列的大模块：

1.  **用户模块**
2.  **拼单模块**
3.  **信任与投诉模块**
4.  **管理模块**

------

##### （1）用户模块

框内可以写小字说明其职责：

-   注册登录
-   实名认证
-   个人资料管理
-   信用分展示

##### （2）拼单模块

框内写：

-   发起拼单
-   浏览与筛选
-   加入与退出
-   支付与退款
-   上传凭证
-   确认送达
-   确认收货
-   状态流转

##### （3）信任与投诉模块

框内写：

-   投诉提交
-   投诉处理
-   信用分变更
-   结果通知

##### （4）管理模块

框内写：

-   管理员登录
-   用户管理
-   拼单管理
-   投诉管理
-   记录查看

------

#### 4.3.3 模块内部结构表现方式

为了和你们后续的程序结构一致，每个核心业务模块内部建议再用三行小字表示其内部采用分层实现：

-   Controller
-   Service
-   Mapper

注意这里**不是必须真的在模块内部画三个嵌套框**，也可以用文字列表表示。
这样做的目的，是让架构图和你们后续采用的 Spring Boot 分层结构保持一致。

#### 不能写成什么

不要写成：

-   API 服务1
-   API 服务2
-   SPI 服务
-   RPC 服务
-   微服务节点

因为这些是老师示例图中的通用企业架构写法，不适合你们当前项目。

------

### 4.4 第四层：数据层

#### 层名称

建议命名为：

**数据层**

#### 层内模块

数据层建议放两个并列大框：

1.  **MySQL 数据库**
2.  **本地文件存储**

------

#### （1）MySQL 数据库

框内可列出数据对象类别：

-   用户数据
-   拼单数据
-   成员关系数据
-   投诉数据
-   信用分记录
-   资金流水
-   通知数据
-   操作日志

这里不要把 10 张表全部塞进去，否则图会太密。
如果组员想细一点，也最多写成：

-   user_account
-   group_order
-   group_order_member
-   complaint
-   capital_record
-   credit_change_record

其余可省略。

------

#### （2）本地文件存储

框内写：

-   订单凭证图片
-   上传文件目录

这部分必须画出来，因为你们系统的订单凭证上传是核心功能，且文件不保存在数据库，而是保存到本地文件系统中。

------

#### （3）Redis 是否绘制

建议：

-   可以**不画**
-   如果要画，只能画成一个虚线小框写“Redis（可选）”

因为你们当前设计口径是 Redis 非必需，不应画成主干架构组件，否则会给老师造成“你们最终实现一定会依赖 Redis”的印象。

------

## 5. 层与层之间的连接关系

为了保证逻辑严谨，图中的连接线建议如下：

### 5.1 访问层 → 前端表示层

三类浏览器都连接到“Vue 前端应用”。

表示：

-   用户通过浏览器访问前端系统页面
-   管理员通过浏览器访问后台页面

### 5.2 前端表示层 → 后端接口层

“Vue 前端应用”连接到“REST API 接口层”。

表示：

-   前端通过 HTTP/HTTPS 调用后端接口

### 5.3 后端接口层 → 业务服务层

“REST API 接口层”连接到“核心业务模块”。

表示：

-   接口层把请求分发给后端业务模块处理

### 5.4 核心业务模块 ↔ 公共支撑模块

建议用内部关联线或依赖箭头表示：

-   用户模块依赖认证鉴权、日志服务
-   拼单模块依赖定时任务、通知服务、日志服务
-   信任与投诉模块依赖通知服务、日志服务
-   管理模块依赖认证鉴权、日志服务

不需要每条都画得特别复杂，但至少要表达出“核心业务模块共享公共支撑能力”。

### 5.5 业务服务层 → 数据层

业务服务层中的各模块统一向下连接到：

-   MySQL 数据库
-   本地文件存储

表示：

-   业务处理依赖数据库持久化
-   凭证上传依赖本地文件系统存储

### 图下注释建议

本图展示校园拼单平台的总体技术结构。系统采用前后端分离的 B/S 架构，用户通过浏览器访问 Vue 前端应用，前端通过 REST API 调用 Spring Boot 后端服务。后端按用户模块、拼单模块、信任与投诉模块、管理模块进行业务划分，并通过认证鉴权、定时任务、通知服务和日志服务提供公共支撑能力。系统底层采用 MySQL 数据库存储业务数据与日志数据，并使用本地文件存储保存订单凭证截图。

## 9. 组员绘图时的注意事项

1.  **不要抄老师示例里的 API Gateway、MQ、RPC、注册中心等结构**
    你们当前项目不是微服务架构，画进去会和最终实现不符。
2.  **不要画成业务流程图**
    架构图只表达层次、模块和依赖关系，不表达“谁先谁后”。
3.  **不要出现未在正文中出现的模块名称**
    比如不要突然加“支付中心”“推荐系统”“风控中心”。
4.  **模块职责必须和概要设计文字一致**
    拼单模块就负责拼单主流程，投诉模块就负责投诉与信用分，不要混乱。
5.  **数据层必须体现本地文件存储**
    因为你们系统有订单凭证上传，这是架构中不可缺的一部分。
6.  **公共支撑模块不要漏掉定时任务**
    因为你们系统的大量关键逻辑依赖自动检查，例如超时取消、自动确认收货、开放投诉。
7.  **图中文字不要太多**
    以模块名为主，职责用短语点到即可，避免整张图塞满句子。
