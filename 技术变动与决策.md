# 技术变动与决策

## 一、技术栈选型

### 1.1 前端技术栈

#### 1.1.1 Vue 3 + Vite
**决策**：使用 Vue 3.5.31 和 Vite 8.0.3

**理由**：
- Vue 3 相比 Vue 2 性能提升显著
- Composition API 提供更好的代码组织方式
- Vite 构建速度快，开发体验优秀
- 生态成熟，组件库丰富
- 学习曲线平缓，适合快速开发

**权衡**：
- 考虑过 React，但 Vue 3 更简洁，适合课程项目
- Vite 虽然较新，但已足够成熟稳定

#### 1.1.2 Element Plus
**决策**：使用 Element Plus 2.13.7

**理由**：
- Vue 3 官方推荐的 UI 组件库
- 组件丰富，覆盖业务需求
- 样式美观，易于定制
- 文档完善，中文友好
- 适合快速开发管理系统

**权衡**：
- 考虑过 Ant Design Vue，但 Element Plus 对 Vue 3 支持更好
- 虽然 Element Plus 体积较大，但课程项目对性能要求不高

#### 1.1.3 Vue Router 5 + Pinia
**决策**：使用 Vue Router 5.0.4 和 Pinia 3.0.4

**理由**：
- Vue Router 是 Vue 官方路由解决方案
- Pinia 是 Vue 3 官方推荐的状态管理库（替代 Vuex）
- Pinia API 更简洁，TypeScript 支持更好
- 与 Vue 3 Composition API 配合良好

**权衡**：
- 考虑过继续使用 Vuex，但 Pinia 更现代
- 两者功能相似，Pinia 学习成本更低

#### 1.1.4 Axios
**决策**：使用 Axios 进行 HTTP 请求

**理由**：
- Promise 支持，API 友好
- 拦截器功能强大，适合统一处理认证
- 浏览器兼容性好
- 社区活跃，生态成熟

**权衡**：
- 考虑过使用 Fetch API，但 Axios 更方便

#### 1.1.5 ESLint + OXLint
**决策**：使用 ESLint 10.1.0 和 OXLint 1.57.0

**理由**：
- ESLint 是主流的 JavaScript 代码检查工具
- OXLint 是较新的工具，速度快
- 两者配合使用，保证代码质量
- 支持 Vue 文件检查

**权衡**：
- 配置多个 linter 可能增加学习成本
- 但对代码质量提升有帮助

### 1.2 后端技术栈

#### 1.2.1 Java 17 + Spring Boot 3
**决策**：使用 Java 17 和 Spring Boot 3.5.13

**理由**：
- Spring Boot 生态成熟，社区活跃
- 约定大于配置，快速开发
- 自动配置，减少繁琐配置
- 内嵌 Tomcat，部署简单
- Java 17 是 LTS 版本，性能优秀
- 适合课程项目快速构建后端服务

**权衡**：
- 考虑过使用 Node.js，但团队成员更熟悉 Java
- Spring Boot 3 虽然较新，但已经稳定
- Java 虽然启动较慢，但课程项目影响不大

#### 1.2.2 Spring Security + JWT
**决策**：使用 Spring Security 和 JWT 认证

**理由**：
- Spring Security 提供完善的安全框架
- JWT 无状态，适合 RESTful API
- 不需要服务端存储 Session
- 支持移动端和 Web 端
- 跨域支持良好

**权衡**：
- JWT 无法主动失效，只能等待过期
- 考虑过使用 Session，但 JWT 更适合前后端分离
- 可以通过刷新 Token 机制解决部分问题

#### 1.2.3 MyBatis Plus
**决策**：计划使用 MyBatis Plus（待添加依赖）

**理由**：
- MyBatis Plus 在 MyBatis 基础上增强，简化开发
- 内置通用 CRUD，减少重复代码
- 支持代码生成器，快速生成实体和 Mapper
- 支持分页插件
- 与 Spring Boot 集成简单

**权衡**：
- 考虑过使用 JPA/Hibernate，但 MyBatis 更灵活
- MyBatis Plus 可能增加学习成本，但提升开发效率

#### 1.2.4 MySQL 8
**决策**：使用 MySQL 8 作为数据库

**理由**：
- 开源免费，使用广泛
- 社区活跃，文档丰富
- 性能优秀，适合中小型应用
- 支持事务，保证数据一致性
- 团队成员熟悉

**权衡**：
- 考虑过使用 PostgreSQL，但 MySQL 更常见
- MySQL 8 相比 5.7 性能提升显著

#### 1.2.5 Lombok
**决策**：使用 Lombok 简化代码

**理由**：
- 减少样板代码（Getter/Setter/Constructor）
- 提高代码可读性
- 与 Spring Boot 集成良好
- 广泛使用，生态成熟

**权衡**：
- Lombok 可能在 IDE 中显示问题
- 需要安装插件支持
- 但整体利大于弊

### 1.3 开发工具

**决策**：要求 Node.js ^20.19.0 || >=22.12.0

**理由**：
- Node.js 20 和 22 是 LTS 版本

**理由**：

- 分支管理功能强大
- 适合多人协作开发
- 行业标准

#### 1.3.3 Maven
**决策**：使用 Maven 管理后端依赖

**理由**：
- Spring Boot 官方推荐
- 依赖管理方便

- 插件生态丰富

## 二、架构决策

### 2.1 前后端分离

**决策**：采用前后端分离架构

**理由**：
- 职责清晰：前端负责展示，后端负责逻辑
- 便于并行开发，提升开发效率
- 前端可独立部署（静态资源）
- 后端可提供多端接口（Web、移动端）
- 技术栈独立，互不影响

**权衡**：
- 需要定义清晰的 API 接口
- 前后端联调需要额外时间
- 跨域问题需要处理

**解决方案**：
- 提前设计 API 文档
- 使用 Mock 数据进行前端开发
- 配置 CORS 处理跨域

### 2.2 RESTful API 设计

**决策**：采用 RESTful API 设计风格

**理由**：
- 接口标准化，易于理解和使用
- 资源导向，语义清晰
- 支持多种数据格式（JSON）
- 无状态，易于扩展
- 适合前后端分离

**权衡**：
- 可能不如 GraphQL 灵活
- 某些场景可能需要多次请求

**解决方案**：
- 合理设计资源嵌套
- 使用查询参数进行过滤和分页
- 必要时提供聚合接口

### 2.3 分层架构

**决策**：后端采用经典分层架构（Controller-Service-Mapper）

**理由**：
- 职责分离，代码清晰
- 易于维护和测试
- 符合 Spring Boot 最佳实践
- 便于扩展功能

**权衡**：
- 可能增加代码量
- 小项目可能显得复杂

**解决方案**：
- 避免过度设计
- 简单业务可适当合并层次

## 三、业务决策

### 3.1 伪支付 / 伪退款

**决策**：使用伪支付和伪退款，不涉及真实资金

**理由**：
- 避免支付系统的复杂性和安全性要求
- 专注于业务逻辑，而非支付集成
- 降低开发成本和风险
- 满足课程项目要求

**权衡**：
- 无法验证真实支付流程
- 可能影响系统可信度

**解决方案**：
- 在文档中明确说明是伪支付
- 设计完整的支付流程模拟
- 重点展示业务逻辑的正确性

### 3.2 本地文件存储

**决策**：使用本地文件系统存储凭证图片

**理由**：
- 实现简单，无需集成云存储
- 适合课程项目规模
- 减少外部依赖
- 部署简单

**权衡**：
- 文件丢失风险
- 无法支持分布式部署
- 文件访问性能较差

**解决方案**：
- 在文档中说明存储方式
- 备份重要文件
- 课程项目规模小，问题不大

### 3.3 简化信用分机制

**决策**：使用简单的信用分计算逻辑

**理由**：
- 复杂的信用算法超出课程范围
- 简单逻辑足以演示功能
- 便于理解和维护

**权衡**：
- 信用分可能不够准确
- 可能存在刷分漏洞

**解决方案**：
- 明确信用分规则
- 管理员可手动调整
- 重点展示功能流程

### 3.4 自动成团和取消

**决策**：实现自动成团和超时取消功能

**理由**：
- 提升用户体验，减少手动操作
- 模拟真实拼单场景
- 展示定时任务能力

**实现方式**：
- 使用 Spring 的 `@Scheduled` 注解
- 定时检查拼单状态
- 达到条件自动更新状态

**权衡**：
- 增加系统复杂度
- 需要考虑并发问题

**解决方案**：
- 使用数据库事务保证一致性
- 合理设置定时任务频率

## 五、开发流程决策

### 5.1 开发顺序

**决策**：优先后端，再前端

**理由**：
- 后端提供 API 接口
- 前端基于接口开发
- 便于前后端联调
- 避免接口变更导致前端返工

**实际流程**：
1. 设计 API 接口
2. 实现后端接口
3. 前端开发（可使用 Mock 数据）
4. 前后端联调
5. 集成测试

### 5.2 版本控制策略

**决策**：使用 Git 分支管理

**分支策略**：
- `main`：主分支，稳定版本
- `develop`：开发分支，集成开发
- `feature/*`：功能分支，开发新功能
- `bugfix/*`：修复分支，修复 Bug
- 个人分支（如 `cyz`）：个人开发

**工作流**：
1. 从 `develop` 创建功能分支
2. 在功能分支上开发
3. 开发完成后合并回 `develop`
4. 定期从 `develop` 合并到 `main`

### 5.3 代码风格

**前端**：
- 使用 ESLint + OXLint 检查
- 遵循 Vue 3 风格指南
- 使用 Prettier 格式化代码

**后端**：
- 遵循 Java 代码规范
- 使用 Lombok 减少样板代码
- 方法命名清晰，注释适当
