# 校园拼单平台需求文档-v6.2
## 1 项目背景
在校园日常生活中，学生之间的拼单行为十分普遍，涵盖餐饮、团购、拼车等多种场景。然而，目前这些拼单需求主要依赖微信群、QQ 群等通用社交平台进行信息发布和沟通。由于缺乏专门为拼单场景设计的工具，信息容易被刷屏淹没，拼单状态难以追踪，沟通成本高，且经常出现放鸽子、转账混乱、取货纠纷等问题，影响了拼单的效率和体验。因此，需要一个专门为校园场景设计的拼单平台，集中管理拼单信息，规范拼单流程，提升学生拼单的整体体验。

## 2 核心问题
当前校园拼单主要存在五个核心问题：一是信息分散，拼单信息零散分布在各个社交群组中，难以集中检索和跟踪状态，用户常常错过合适的拼单机会；二是信任缺失，缺乏用户信用记录和约束机制，经常出现参与者临时放鸽子、发起人失联或取货后不确认等行为，影响拼单成功率；三是资金混乱，拼单涉及多人转账，谁付了钱、谁还没付没有统一记录方式，容易产生纠纷；四是交付不便，取货地点和时间不明确，容易出现冒领或长时间无人取货的问题；五是纠纷无门，当出现放鸽子、资金纠纷等问题时，用户没有正式的投诉和处理渠道，只能自行协调，往往难以解决。

## 3 解决方案概述
本平台通过构建统一的拼单信息发布与全流程管理机制，系统性地解决上述问题，精简冗余功能，确保流程简洁、可行且无真实资金风险。平台提供拼单的发起、加入、状态跟踪、自动成团、超时取消等核心功能，并围绕信任、资金、交付三大关键环节设计配套机制：

- 信任机制：采用信用分体系（极简规则），约束用户违规行为（如发起人未上传凭证、被投诉成立等），信用分仅做展示和记录，暂不设置功能限制；简化实名认证流程，绑定学号保障校园用户身份真实性。
- 资金机制：平台采用代收代付模式，不涉及真实资金流转。用户加入拼单时按预估人均金额完成支付，款项由平台代收；所有参与者（含发起人）支付完成后自动成团；发起人购买商品后上传凭证及实际总金额，当所有参与者确认收货后，平台将实际总金额打款给发起人。若实际金额低于预估金额，系统自动将差额退回给所有参与者（含发起人）；若实际金额高于预估金额，超出部分由发起人自行承担或线下补收，平台不作自动补款。未成团用户可自行退出并自动退款，已成团禁止退出。
- 交付机制：平台不强制介入取货验证，支付为确认收货的前置条件。发起人上传凭证时需填写实际总金额和预计送达时间区间；商品实际送达后，发起人点击"商品已送达"方可开启确认收货倒计时。参与者取货后点击"确认收货"完成交付闭环；若参与者30分钟内未点击"确认收货"，系统默认完成收货。若发起人超过"预计最晚送达时间+60分钟"仍未点击"商品已送达"，参与者可发起投诉，由管理员处理后批量退款。同时，通过投诉机制和信用分作为事后约束，保障交付环节的可靠性。
- 纠纷处理：简化投诉类型，支持“发起人未购买商品”（超时未上传凭证自动触发）和“发起人上传虚假凭证”两类投诉。用户填写简单说明即可发起投诉，管理员一键处理：为所有参与者批量退款、扣除发起人信用分，确保纠纷可追溯、可解决。

通过上述机制的组合，平台在保持流程简洁的同时，建立起完整的信任保障、资金安全和纠纷处理能力，适配两周开发周期，满足课程项目验收标准。

## 4 系统组成
本平台由以下核心模块组成，剔除复杂模块，聚焦核心功能，确保开发可行性：

|模块|职责|涉及角色|
|---|---|---|
|用户模块|负责用户注册、登录、实名认证、个人资料管理、信用分展示|普通用户、管理员|
|拼单模块|负责拼单的全流程管理，包括发起、加入、状态跟踪、自动成团、超时取消、伪支付与伪退款、凭证上传（含实际总金额和预计送达时间区间）、确认送达、多退少补（退款自动、补款手动）、确认收货（含30分钟自动确认）、确认收货后平台打款给发起人；未成团可自行退出，已成团禁止退出|普通用户|
|信任模块|负责信用分计算与更新，以及各类简单易处理投诉提交与处理，不新增复杂评价功能|普通用户、管理员|
|管理模块|负责管理员后台的用户管理、拼单管理、投诉处理、交易及退款记录查看|管理员|

### 5.2 拼单模块
#### 5.2.1 发起人视角功能
|阶段|功能点|优先级|说明|
|---|---|---|---|
|发起|填写拼单信息|P0|商品名称、总人数、预估总金额、取货点、截止时间；仅支持AA制；系统自动计算并显示预估人均金额（预估总金额 ÷ 总人数）；可填写商品描述（非必填），总人数包含发起人本人|
|发起|发布拼单|P0|发布后拼单进入“拼单中”状态；发起人需要支付（作为参与者之一），发布后系统自动将发起人加入参与者列表，发起人需调用伪支付接口完成预估人均金额支付|
|成团|收到成团 / 超时取消通知|P0|系统自动推送状态变更通知|
|购买|上传订单凭证|P0|购买后上传订单截图（佐证实际金额真实性），同时填写实际总金额和预计送达时间区间（如“12:30-12:50”）；已成团后30分钟内必须上传，超时自动开放投诉通道|
|支付|查看参与者支付状态|P0|查看每个参与者的支付状态（未支付 / 已支付）；所有参与者均需完成支付，全部支付完成且人数满员方可成团；|
|取货|确认送达|P0|商品实际送达取货点后，发起人点击“商品已送达”，系统通知所有参与者可取货，并开启30分钟确认收货倒计时|
|取货|等待参与者确认收货|P0|参发起人点击“确认送达”后，参与者可点击确认收货，发起人可查看确认状态、确认时间；若参与者30分钟内未点击"确认收货"，系统默认其已收货，自动更新确认状态|
|取货|收到所有确认收货通知|P1|所有参与者（含默认确认收货）确认后，系统自动将平台代收的总金额打款给发起人，拼单状态变更为“已完成”|

#### 5.2.2 参与者视角功能
|阶段|功能点|优先级|说明|
|---|---|---|---|
|浏览|查看拼单列表|P0|按状态（拼单中 / 已成团 / 已完成）分类展示，支持排序；列表展示核心信息（商品名称、剩余名额、实际人均、取货点），方便快速筛选|
|浏览|搜索 / 筛选拼单|P1|按商品关键词、取货点筛选拼单（简化筛选逻辑，不新增复杂条件）|
|加入|查看拼单详情|P0|展示拼单信息、参与者列表、应付预估人均金额、支付状态、取货信息、发起人信息；可查看拼单创建时间、剩余时间，辅助判断是否加入|
|加入|填写备注|P0|点击加入拼单时填写备注（如 “七分糖、少冰”），非必填；显示在拼单详情页参与者列表，所有成员可见|
|加入|加入拼单|P0|确认加入后，点击“支付”，调用伪支付接口完成预估人均金额支付；支付金额由平台代收，支付成功后占用拼单名额；总人数包含发起人，发起人也需支付|
|成团|收到成团 / 超时取消通知|P0|系统自动推送状态变更通知|
|多退少补|查看差额|P0|发起人上传凭证后，系统展示：预估人均、实际人均、差额（预估−实际）；仅当实际金额 < 预估金额时触发退款；实际金额 > 预估金额时不处理（发起人自行承担或线下补收）|
|多退少补|退款|P0|当实际金额 < 预估金额时，系统后端自动调用伪退款接口，将差额从平台退回给用户，无需用户操作|
|退出|拼单退出|P0|仅拼单中（未成团）可自行退出，系统调用伪退款接口，返回“退款成功”即完成退款，重置支付状态、释放名额；已成团状态禁止退出，无退出入口|
|取货|点击确认收货|P0|发起人点击“商品已送达”后，参与者前往取货点取货，点击确认收货完成交付闭环；确认后无法撤销，系统同步更新拼单状态，通知发起人；若30分钟内未点击，系统默认完成收货|
|完成|收到拼单完成通知|P1|所有人确认收货（含默认确认）后，系统通知参与者拼单完成，可查看拼单完整交易记录，用于后续核对|

#### 5.2.3 系统自动功能
|功能点|优先级|说明|
|---|---|---|
|满员自动成团|P0|当加入人数达到设定总人数（且所有参与者均已完成支付）时，系统自动锁定拼单，状态变更为 “已成团”，通知所有参与者|
|超时自动取消|P0|截止时间到但未满员时，系统自动取消拼单，通知所有参与者，并调用伪退款接口为已支付用户办理退款|
|金额计算|P0|统一价模式：自动计算预估人均（预估总金额÷总人数）和实际人均（实际总金额÷总人数）；不支持自定义价模式|
|多退少补-自动退款|P0|当实际总金额 < 预估总金额时，后端自动计算差额（预估人均 − 实际人均），为所有参与者（含发起人） 调用伪退款接口退回差额|
|多退少补-超额处理|P0|不处理。当实际总金额 > 预估总金额时，超出部分由发起人自行承担或线下向参与者补收，平台不介入|
|超时检测（凭证上传）|P0|已成团后30分钟内，发起人未上传凭证，系统自动开放投诉通道，弹窗提醒所有参与者|
|确认收货超时自动处理|P0|发起人点击“商品已送达”后，若参与者30分钟内未点击"确认收货"，系统自动默认其已收货，更新确认收货状态，并同步通知发起人|
|超时送达投诉触发|P1|发起人超过“预计最晚送达时间+60分钟”仍未点击“商品已送达”时，系统自动开放投诉通道，参与者可发起投诉（类型：发起人未购买商品）|

#### 5.2.4 状态定义
1. 拼单状态（5种）

|状态名称|说明|进入条件|退出条件|
|---|---|---|---|
|拼单中|拼单已发布，正在等待参与者加入|发起人发布拼单成功|①满员且全部支付 → 已成团<br/>②截止时间到未满员 → 已取消|
|已成团|拼单已满员且全部支付成功，等待发起人购买并上传凭证|满员且所有参与者完成支付|①发起人上传凭证 → 待收货<br/>②发起人超时30分钟未上传凭证 → 已取消|
|待送达|发起人已上传凭证（含实际总金额和预计送达时间），等待商品送达取货点|发起人上传凭证成功|①发起人点击“商品已送达” → 待收货<br/>②超过“预计最晚送达时间+60分钟”未送达 → 开放投诉通道（拼单暂不取消，投诉成立后取消）|
|待收货|发起人已上传凭证，等待参与者取货确认|发起人上传凭证成功|所有参与者确认收货 → 已完成|
|已完成|所有参与者均已确认收货，拼单正常结束|所有参与者确认收货（含自动确认）|无（终态）|
|已取消|拼单提前终止，未正常完成|①截止时间到未满员<br/>②发起人超时30分钟未上传凭证<br/>③管理员强制取消|无（终态）|

2. 用户-拼单状态（5种）

用户在某个具体拼单中的参与状态。

|状态名称|说明|进入条件|退出条件|
|---|---|---|---|
|待支付|用户已点击加入，但未完成支付（瞬时状态）|用户点击“加入拼单”|完成支付 → 已支付|
|已支付|用户已完成支付，等待成团|调用伪支付接口成功|①成团 → 待收货<br/>②拼单取消 → 已退款<br/>③用户主动退出 → 已退款|
|待收货|已成团且发起人已上传凭证，等待用户取货确认|拼单状态变为“待收货”|确认收货 → 已收货|
|已收货|用户已确认收货（或系统自动确认）|用户点击确认收货 或 20分钟自动确认|无（终态）|
|已退款|拼单取消或用户主动退出后，已收到退款|系统调用伪退款接口成功|无（终态）|

3. 投诉单状态（2种）

|状态名称|说明|进入条件|退出条件|
|---|---|---|---|
|待处理|投诉已提交，等待管理员处理|用户提交投诉成功|管理员处理 → 已处理|
|已处理|管理员已完成处理（退款+扣分）|管理员执行固定处理后|无（终态）|

4. 用户账号状态（2种）

|状态名称|说明|进入条件|退出条件|
|---|---|---|---|
|正常|用户可以正常使用所有功能|用户注册成功|管理员封禁 → 已封禁|
|已封禁|用户被管理员封禁，无法登录和使用任何功能|管理员执行封禁操作|管理员解封 → 正常|

### 5.3 信任模块
|功能点|优先级|说明|
|---|---|---|
|互相评价|P2|暂不实现，简化流程，聚焦核心功能开发|
|信用分计算|P1|极简规则：初始 80 分；完成 1 次有效拼单 +2 分；投诉成立 -10 分；主动放鸽子 -15 分；发起人未上传凭证被核实 -10 分（额外扣分）|
|信用分阈值限制|P2|暂不实现，简化流程，仅展示信用分，不限制用户核心操作|
|提交投诉|P1|支持"发起人未购买商品"类投诉，触发条件包括：①成团后30分钟内未上传凭证；②上传凭证后超过“预计最晚送达时间+60分钟”仍未点击“商品已送达”。支持"发起人上传虚假凭证（凭证与拼单商品不符）"类投诉。|
|投诉处理|P1|管理员只需查看说明和投诉类型，一键执行固定处理（两类投诉处理方式统一）：调用伪退款接口为所有参与者批量退款（伪接口返回“退款成功”即完成），扣除发起人10分信用分|
|投诉结果通知|P1|系统将处理结果通知投诉方和被投诉方|

**信用分规则说明**
- 新用户注册后默认信用分为 80 分（满分 100 分）
- 完成拼单可加分；放鸽子、发起人未上传凭证、被投诉成立等行为会扣分
- 信用分仅在个人主页展示，暂不设置阈值限制（简化流程）
- 用户可在个人主页查看当前信用分及扣分、加分记录

## 6 核心流程说明
### 6.1 发起人流程
|步骤|操作|状态变化|
|---|---|---|
|1|注册登录 + 实名认证|-|
|2|填写拼单信息（商品名称、总人数、预估总金额、取货点、截止时间），系统自动计算预估人均金额，总人数包含发起人本人|-|

|12|若发起人超过“预计最晚送达时间+60分钟”仍未点击“商品已送达”，系统自动开放投诉通道，参与者可发起投诉，管理员处理后批量退款并扣分|拼单状态 → 已取消；<br/>所有参与者的用户-拼单状态 → 已退款|

### 6.4 投诉处理流程（2类投诉通用基础流程）
**触发条件**
- 类型1（发起人未购买商品）包括以下两种情况：
情况A：拼单已成团后，发起人未在30分钟内上传订单凭证，系统自动开放投诉通道；
情况B：发起人已上传凭证，但超过“预计最晚送达时间+60分钟”仍未点击“商品已送达”，系统自动开放投诉通道。
- 类型2（发起人上传虚假凭证）：发起人上传的订单凭证与拼单商品信息不符，参与者发现后可主动发起投诉。

|操作|参与方|说明|
|---|---|---|
|用户选择投诉类型，填写简要说明（可选上传证据）|投诉方|用户仅需选择2类投诉类型中的一种，填写简单投诉说明（无需复杂描述，1-2句话说明情况即可），无需上传任何证据，系统自动关联拼单信息，生成“待处理”投诉单，大幅降低开发和使用成本，适配两周开发周期。（【状态变化】 投诉单状态 → 待处理）|
|系统推送投诉提醒至管理员后台|系统|仅推送核心信息（投诉类型、关联拼单ID），无需复杂标识，简化通知逻辑|
|查看投诉类型、拼单信息，一键执行固定处理|管理员|仅查看投诉方填写的简单说明，无需核实任何证据、无需联系任何一方，根据投诉类型点击一键执行固定处理：调用伪退款接口为所有参与者（含发起人）批量退款（退回已支付的预估金额）、扣除发起人信用分<br/>（【状态变化】 投诉单状态 → 已处理；拼单状态 → 已取消；所有参与者的用户-拼单状态 → 已退款）|
|系统自动推送标准化处理结果|系统|采用标准化模板通知（例：“您的投诉已处理，发起人扣10分，已为您退款”），无需手动编辑，自动更新投诉状态为“已处理”|

## 8 项目迭代规划（总周期：两周）
### 8.1 迭代总体说明
总周期：14 天（2 周）
迭代次数：2 次迭代，每次迭代 7 天

### 8.2 第一次迭代（第 1-7 天：核心功能开发，实现基础可用）
#### 迭代目标
完成核心模块基础功能开发，实现用户注册登录、拼单发起与加入、伪支付与伪退款、自动成团 / 超时取消等核心流程，确保平台可正常运行，无关键功能缺失。

#### 核心开发任务
1. 用户模块：完成注册、登录、实名认证核心功能，实现信用分初始赋值（80 分）及展示功能；
2. 拼单模块：完成拼单信息填写与发布、拼单列表浏览、拼单详情查看、加入拼单、自动成团、超时自动取消、未成团退出与退款功能；
3. 基础交互：完成各模块页面基础布局，确保页面跳转流畅，操作反馈及时；
4. 需求组：完成每日需求澄清，同步开发疑问，制定核心功能验收标准。

#### 验收标准
- 用户可正常完成注册、登录、实名认证，信用分初始为 80 分并可正常展示；
- 发起人可正常填写并发布拼单，参与者可正常浏览、查看、加入拼单并完成支付；
- 未成团可正常退出并退款，拼单可自动成团与自动取消，状态与通知可正常更新。

### 8.3 第二次迭代（第 8-14 天：功能完善 + 纠纷处理 + 测试优化）
#### 迭代目标
完善凭证上传、确认收货、投诉处理及管理后台功能，完成全流程测试与优化，确保平台流程闭环、稳定可用。

#### 核心开发任务
1. 拼单模块：完成订单凭证上传、确认收货、20 分钟自动确认收货及拼单全状态更新；
2. 信任模块：实现信用分规则、投诉提交与处理功能；
3. 管理模块：完成管理员登录、用户管理、拼单管理、投诉处理、记录查看功能；
4. 测试与优化：开展全流程测试，修复问题并优化交互体验；
5. 文档完善：整理项目文档，确保内容完整可追溯。

#### 验收标准
- 发起人可正常上传凭证（含实际总金额和预计送达时间区间），可正常点击“商品已送达”，参与者可正常确认收货，30分钟超时可自动确认；
- 超过“预计最晚送达时间+60分钟”发起人仍未点击“商品已送达”时，系统自动开放投诉通道，管理员可正常处理；
- 用户可正常提交投诉，管理员可完成处理并同步结果，信用分可正常更新；
- 管理员可正常使用后台全部核心功能，异常拼单可强制取消并退款；
- 平台无明显功能 bug，流程闭环，交互流畅，满足两周开发与项目验收要求。
