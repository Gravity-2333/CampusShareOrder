## 各层职责

| 层 / 包              | 最终职责                                                  |
| ------------------ | ----------------------------------------------------- |
| `config`           | Spring 配置、Jackson 时间格式、MyBatis-Plus、资源映射、JWT/文件存储配置绑定 |
| `common.response`  | 统一返回体 `ApiResponse<T>`、统一分页体 `PageResult<T>`          |
| `common.exception` | 统一业务异常、全局异常处理                                         |
| `common.enums`     | 所有状态、角色、错误码、记录类型枚举                                    |
| `common.util`      | JWT、编号生成、当前登录人获取、金额处理、时间处理、脱敏工具                       |
| `controller`       | 只接收请求、校验参数、调用 Service、返回统一 JSON                       |
| `dto`              | 所有入参对象，字段名与前端 JSON 完全一致                               |
| `vo`               | 所有出参对象，字段名与前端 JSON 完全一致                               |
| `entity`           | 与数据库表一一对应的持久化对象                                       |
| `mapper`           | MyBatis-Plus Mapper，负责数据库 CRUD / 自定义 SQL              |
| `security`         | JWT 解析、认证注入、未登录/无权限处理                                 |
| `service`          | 业务接口，定义模块能力                                           |
| `service.impl`     | 业务实现，所有状态流转、退款、投诉、日志、通知都写在这里                          |
| `task`             | 只放 `@Scheduled` 任务，不放业务细节；任务内部调用 Service              |

### 冲突修正说明

1. 当前仓库的 `scheduler` 包统一更名为 `task`。
2. 当前仓库的 `common/api.java` 统一拆为 `common/response/ApiResponse.java`。
3. 当前仓库的 `service/impl.java` 必须删除，改成真实的 `*ServiceImpl.java`。
4. 后端保持**一个** Spring Boot 项目，不拆多项目。

# 2. 最终 Entity 设计

系统核心实体固定为 10 个：`UserAccount`、`AdminAccount`、`GroupOrder`、`GroupOrderMember`、`OrderReceipt`、`Complaint`、`CapitalRecord`、`CreditChangeRecord`、`Notification`、`OperationLog`。字段范围、表范围、状态字段拆分、唯一约束、索引方向，全部与系统设计文档、开发规划和前端冻结契约对齐。  

## 2.1 不直接入库、只做聚合返回的字段总表

| 前端字段                                | 来源                                         | 是否数据库直接字段 |
| ----------------------------------- | ------------------------------------------ | --------- |
| `viewerRoleInOrder`                 | 当前用户 + 订单创建人 + 成员关系聚合                      | 否         |
| `creatorNickname`                   | `user_account.nickname` 关联查询               | 否         |
| `remainingCount`                    | `totalMemberCount - currentMemberCount`    | 否         |
| `currentUserJoined`                 | 当前用户是否存在 `group_order_member`              | 否         |
| `currentUserPayStatus`              | 当前用户成员行 `pay_status`                       | 否         |
| `initiatorInfo.phoneMasked`         | `user_account.phone` 脱敏                    | 否         |
| `initiatorInfo.studentNoMasked`     | `user_account.student_no` 脱敏               | 否         |
| `paymentSummary.*` 中统计字段            | `group_order_member` / `capital_record` 聚合 | 否         |
| `receiveInfo.autoConfirmDeadlineAt` | `deliveredAt + 30 分钟`                      | 否         |
| `timeline`                          | `operation_log` + 系统拼装                     | 否         |

### 冲突修正说明

1. `UserAccount` **不保留** `role` 字段；普通用户与管理员分表，`role` 只在 JWT 与 `GET /api/auth/me` 中返回。
2. `GroupOrder.estimatedPerAmount`、`actualPerAmount` **作为数据库字段保留**，由后端在创建订单 / 上传凭证时计算写入；不再把它们视为纯聚合字段。
3. `Complaint` 增加唯一约束 `group_order_id + complainant_user_id`，V1 冻结为“**同一用户对同一订单只能投诉一次**”。
4. `CreditChangeRecord.reasonType` V1 只落两类：`ORDER_FINISHED`、`COMPLAINT_CONFIRMED`；不单独落 `NO_RECEIPT_UPLOAD` 额外扣分，避免与“投诉成立扣 10 分”重复扣分。

# 3. DTO / VO 冻结表

DTO / VO 的字段名必须与前端 JSON 名称完全一致，统一使用 `lowerCamelCase`；成功响应统一包装为 `{ code, message, data }`，分页统一包装为 `{ list, page, pageSize, total, pages }`。当前前端已经冻结接口命名、`OrderDetailVO` 模块结构、`actionFlags` 思路、状态名、测试口径，因此后端 DTO / VO 不能再自创字段或改路径。

2. `ComplaintInfoVO` **不保留** `canComplain`，按钮控制唯一来源是 `actionFlags.canCreateComplaint`。
3. 管理端订单详情直接复用 `OrderDetailVO`；当 `viewerRoleInOrder=ADMIN` 时，所有可写动作 flag 固定为 `false`，`canViewReceipt` 仅随 `hasReceipt` 决定。
4. `GET /api/complaints/{complaintId}` 保留，供“我的投诉”抽屉读取使用；它不是独立页面接口，但它是正式只读接口。

3. `40303` 只用于**需要实名的业务接口**，不用于 `/api/auth/me`。

# 6. 安全与鉴权冻结方案

V1 只使用 `Spring Security + JWT + 数据库二次校验`；不使用 Session，不使用 MQ，不做复杂授权模型。后台与普通用户分离登录，JWT 中必须带角色，所有受保护请求在进入业务前都要用数据库确认账号状态。 

## 6.1 JWT 载荷规则

| 账号类型    | claims 最终版                                   |
| ------- | -------------------------------------------- |
| `USER`  | `sub`, `role=USER`, `userId`, `iat`, `exp`   |
| `ADMIN` | `sub`, `role=ADMIN`, `adminId`, `iat`, `exp` |

**冻结规则**：

* JWT 只作为身份入口，不作为实名状态和封禁状态的最终真相。
* 每次受保护请求都要根据 `userId/adminId` 重新查库。

## 6.2 接口权限表

| 权限级别                            | 接口                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| ------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `permitAll`                     | `POST /api/auth/register`、`POST /api/auth/login`、`POST /api/auth/admin/login`、`GET /uploads/**`                                                                                                                                                                                                                                                                                                                                                                                                      |
| `authenticated (USER or ADMIN)` | `GET /api/auth/me`、`POST /api/auth/logout`                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| `USER` 且可未实名                    | `POST /api/users/verify-student`                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| `USER` 且必须实名                    | `GET /api/users/profile`、`PUT /api/users/profile`、`GET /api/users/credit`、`GET /api/users/my-orders`、`GET /api/orders`、`POST /api/orders`、`GET /api/orders/{orderId}`、`POST /api/orders/{orderId}/join`、`POST /api/orders/{orderId}/pay`、`POST /api/orders/{orderId}/exit`、`POST /api/orders/{orderId}/upload-receipt`、`POST /api/orders/{orderId}/mark-delivered`、`POST /api/orders/{orderId}/confirm-received`、`POST /api/complaints`、`GET /api/complaints/my`、`GET /api/complaints/{complaintId}` |
| `ADMIN`                         | `/api/admin/**` 全部                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |

## 6.3 Spring Security 组织规则

| 项目         | 最终规则                                                                   |
| ---------- | ---------------------------------------------------------------------- |
| CSRF       | 关闭                                                                     |
| Session    | `STATELESS`                                                            |
| 表单登录       | 关闭                                                                     |
| HTTP Basic | 关闭                                                                     |
| JWT 过滤器    | `JwtAuthenticationFilter` 放在 `UsernamePasswordAuthenticationFilter` 之前 |
| 未登录处理      | `RestAuthenticationEntryPoint` 返回 `40101/40102` JSON                   |
| 无权限处理      | `RestAccessDeniedHandler` 返回 `40301` JSON                              |
| CORS       | 统一在 `CorsConfig` 配置，允许前端开发域名                                           |
| 资源映射       | `GET /uploads/**` 放行，供凭证图片直接访问                                         |
| Principal  | 统一封装为 `LoginIdentity`，包含 `role`、`userId/adminId`                       |

## 6.4 `GET /api/auth/me` 恢复身份规则

| token 角色 | 后端行为                                                            |
| -------- | --------------------------------------------------------------- |
| `USER`   | 解析 `userId`，查 `user_account`，校验存在、未封禁，返回 `CurrentLoginInfoVO`   |
| `ADMIN`  | 解析 `adminId`，查 `admin_account`，校验存在、未封禁，返回 `CurrentLoginInfoVO` |

## 6.5 被封禁用户处理规则

| 场景                                     | 后端返回    |
| -------------------------------------- | ------- |
| 被封禁普通用户再次登录                            | `40302` |
| 被封禁管理员再次登录                             | `40302` |
| 已持有旧 token 的被封禁用户访问任何受保护接口             | `40302` |
| 已持有旧 token 的被封禁用户访问 `GET /api/auth/me` | `40302` |

### 冲突修正说明

1. `GET /uploads/**` 必须 `permitAll`，否则前端 `<img>` 无法直接加载凭证图。
2. 认证通过后仍要查库，不能只相信 token 里的 `status/isVerified` 快照。
3. 后端实名校验与前端路由守卫保持一致：**只要进入订单/投诉/信用分页/我的拼单/个人中心这些业务接口，就必须实名**。

# 7. 状态流转与事务边界

创建订单、加入、支付、成团、退出、上传凭证、差额退款、确认送达、确认收货、自动确认、自动完成、投诉、管理员取消和处理投诉，都必须按事务边界执行；开发规划与系统设计文档已明确“自动逻辑统一放在服务层或定时任务，不要散写在控制器里”，并要求支付、退款、投诉、信用分、日志都可追溯。 

## 7.1 事务规则表

| 加入订单    | `group_order_member` `group_order` `operation_log`                                                                    | 新增成员；`current_member_count +1`                                                                            | 否                          | 是，`ORDER_JOIN`                                     | 否                   | 否                                      | 是              |

| 投诉提交    | `complaint` `operation_log`                                                                                           | 新增投诉 `PENDING`                                                                                            | 否                          | 是，`COMPLAINT_CREATE`                               | 否                   | 否                                      | 是              |
| 投诉处理    | `complaint` `group_order` `group_order_member` `capital_record` `credit_change_record` `operation_log` `notification` | 投诉 `PROCESSED`；订单 `CANCELED`；成员 `ACTIVE->CANCELED`；发起人扣分                                                  | 是，`REFUND_CANCEL` 净退款      | 是，`COMPLAINT_HANDLE`、`ORDER_CANCELED_BY_COMPLAINT` | 是，投诉结果 & 订单取消通知     | 是，发起人 `COMPLAINT_CONFIRMED -10`        | 是              |
| 管理员取消订单 | `group_order` `group_order_member` `capital_record` `operation_log` `notification`                                    | 订单 `CANCELED`；成员 `ACTIVE->CANCELED`                                                                       | 是，`REFUND_CANCEL` 净退款      | 是，`ORDER_CANCELED_BY_ADMIN`                        | 是                   | 否                                      | 是              |

## 7.2 关键实现规则补充

| 规则              | 最终版                                                                       |
| --------------- | ------------------------------------------------------------------------- |
| 发起人是否也必须支付      | 必须                                                                        |
| 成团条件            | `currentMemberCount == totalMemberCount` 且全部 `ACTIVE` 成员 `payStatus=PAID` |
| 发起人能否退出自己的拼单    | 不能，固定返回 `40912`                                                           |
| 差额退款何时发生        | 只在上传凭证时                                                                   |
| 差额退款是否改变主状态     | 不改变，上传成功后仍进入 `WAIT_DELIVERY`                                              |
| 取消退款公式          | `refundCancelAmount = max(payAmount - refundAmountTotal, 0)`              |
| 完成结算公式          | `settleToCreatorAmount = min(actualTotalAmount, estimatedTotalAmount)`    |
| 投诉成立后订单状态       | 固定 `CANCELED`                                                             |
| 自动确认收货后如何完成     | 调 `tryCompleteOrder(orderId)` 立即判定是否 `COMPLETED`                          |
| 发起人确认送达时自己的收货状态 | 直接置 `RECEIVED`                                                            |

### 冲突修正说明

1. **发起人不能退出自己创建的拼单**；这条在前端按钮矩阵里隐含，但必须在后端硬校验。
2. **订单完成后的信用分规则固定为：全部 `ACTIVE` 成员 +2；投诉成立时被投诉发起人 -10**。不实现“未上传凭证再额外 -10”双重扣分。
3. `current_member_count` 只在 `join` / `exit` 时改；订单被取消时不归零，保留取消当时的人数快照。

# 8. 定时任务冻结方案

当前版本只允许使用 `@Scheduled + Service + @Transactional`；不引入 MQ。自动规则的扫描条件、频率、服务调用和日志动作全部固定如下。 

## 8.1 最终任务清单

| 所属类              | 方法名                               | 扫描条件                                                                                                                                                       | 扫描频率   | 调用 Service                                     | 修改表                                                                                                           | 写日志                                         |
| ---------------- | --------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------- | ------ | ---------------------------------------------- | ------------------------------------------------------------------------------------------------------------- | ------------------------------------------- |
| `OrderScheduler` | `cancelExpiredOpenOrders()`       | `group_order.status=OPEN AND deadline_at <= now`                                                                                                           | 每 1 分钟 | `OrderService.cancelExpiredOpenOrders()`       | `group_order` `group_order_member` `capital_record` `notification` `operation_log`                            | `ORDER_AUTO_CANCELED`                       |
| `OrderScheduler` | `openReceiptTimeoutComplaints()`  | `group_order.status=GROUPED AND complaint_opened=0 AND receipt_upload_deadline_at <= now AND 不存在 order_receipt`                                            | 每 1 分钟 | `OrderService.openReceiptTimeoutComplaints()`  | `group_order` `notification` `operation_log`                                                                  | `COMPLAINT_CHANNEL_OPENED_RECEIPT_TIMEOUT`  |
| `OrderScheduler` | `openDeliveryTimeoutComplaints()` | `group_order.status=WAIT_DELIVERY AND complaint_opened=0 AND delivered_at IS NULL AND expected_delivery_end_at + 60m <= now`                               | 每 1 分钟 | `OrderService.openDeliveryTimeoutComplaints()` | `group_order` `notification` `operation_log`                                                                  | `COMPLAINT_CHANNEL_OPENED_DELIVERY_TIMEOUT` |
| `OrderScheduler` | `autoConfirmReceivedMembers()`    | `group_order.status=WAIT_RECEIVE AND group_order_member.join_status=ACTIVE AND is_creator=0 AND receive_status=WAIT_CONFIRM AND delivered_at + 30m <= now` | 每 1 分钟 | `OrderService.autoConfirmReceivedMembers()`    | `group_order_member`，必要时 `group_order` `capital_record` `credit_change_record` `notification` `operation_log` | `ORDER_AUTO_RECEIVED`、必要时 `ORDER_COMPLETED` |
| `OrderScheduler` | `recoverCompletedOrders()`        | `group_order.status=WAIT_RECEIVE` 且所有 `ACTIVE` 成员 `receive_status in (RECEIVED,AUTO_RECEIVED)`                                                             | 每 1 分钟 | `OrderService.recoverCompletedOrders()`        | `group_order` `capital_record` `credit_change_record` `notification` `operation_log`                          | `ORDER_COMPLETED`                           |

## 8.2 任务实现规则

| 规则      | 最终版                                              |
| ------- | ------------------------------------------------ |
| 定时任务类数量 | 只保留 `OrderScheduler`                             |
| 定时器实现   | `@Scheduled(cron = "0 */1 * * * ?")`             |
| 事务位置    | 任务方法本身不写业务，调用 `@Transactional` 的 Service 方法      |
| 幂等控制    | 退款、结算前必须先查 `capital_record.biz_no` 或同类记录是否已存在    |
| 扫描分页    | 每次按固定批量查询，例如 100 条，循环处理                          |
| 日志      | 所有任务型动作都写 `operation_log`，`operator_type=SYSTEM` |
| 通知      | 统一写入 `notification` 表，不做 WebSocket、不做消息队列        |

### 冲突修正说明

1. V1 **不新建** `ComplaintScheduler`；投诉开放任务仍然是订单超时任务。
2. “成团后 30 分钟未上传凭证”和“预计最晚送达 + 60 分钟未确认送达”都**只开放投诉，不自动取消订单**。
3. `recoverCompletedOrders()` 是**兜底任务**，不是主触发路径；主触发仍然是 `confirm-received` 和 `autoConfirmReceivedMembers()` 内部的 `tryCompleteOrder(orderId)`。

# 10. 最小后端开发顺序

开发顺序必须和前端主链路一致，先跑通认证与订单 P0，再补闭环，再补投诉和后台；开发规划文档已经明确要求先补依赖、先冻结数据库、再做认证、用户、拼单主流程接口。 

| 步骤               | 本步产出                                                                                                                                                                                    | 前端可联调页面                 |
| ---------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------- |
| 第一步：统一基础设施       | `application.yml`、MySQL 连接、MyBatis-Plus、统一返回体、统一异常、全部枚举、JWT 工具、Security 骨架、`GET /uploads/**` 资源映射、基础表 SQL                                                                               | 暂无正式页面；可先保后端可启动         |
| 第二步：Auth         | `user_account` / `admin_account`、`POST /api/auth/register`、`POST /api/auth/login`、`POST /api/auth/admin/login`、`GET /api/auth/me`、`POST /api/auth/logout`                               | 登录页、注册页、管理员登录页          |
| 第三步：User         | `POST /api/users/verify-student`、`GET/PUT /api/users/profile`、`GET /api/users/credit`、`GET /api/users/my-orders`（先可空列表）                                                                 | 实名认证页、个人中心页、信用分页        |
| 第四步：Order P0 主链路 | `group_order`、`group_order_member`、`capital_record`；`POST /api/orders`、`GET /api/orders`、`GET /api/orders/{orderId}`、`POST /join`、`POST /pay`、`POST /exit`；`tryGroupAfterPay()`；未成团取消任务 | 拼单列表页、拼单详情页、发起拼单页、我的拼单页 |
| 第五步：收尾闭环         | `order_receipt`、`POST /upload-receipt`、`POST /mark-delivered`、`POST /confirm-received`、差额退款、自动确认收货、自动完成、结算、信用分 +2                                                                       | 订单详情页完整动作联调             |
| 第六步：投诉与后台        | `complaint`、`credit_change_record`、`notification`、`operation_log`；投诉接口、管理员用户/订单/投诉/记录接口、管理员取消订单、处理投诉                                                                                    | 投诉页、管理后台全部页面            |

### 冲突修正说明

1. `GET /api/users/my-orders` 在第三步就先做，哪怕先返回空分页，也要让前端页面先接上。
2. 第四步必须把 `GET /api/orders` 和 `GET /api/orders/{orderId}` 一起做出来，否则前端 P0 页面无法真正联调。
3. 差额退款、自动确认收货、结算都放在第五步，不拆到后台模块之后。

# 11. 后端验收用例最小集合

P0 验收与前端冻结契约完全对齐：注册登录、实名认证、创建订单、加入订单、支付、自动成团、未成团超时取消、订单列表、订单详情。 

| 用例编号               | 前置条件                                      | 调用接口                                                                      | 预期数据库变化                                                                                             | 预期返回结构                                                                                                                      |
| ------------------ | ----------------------------------------- | ------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------- |
| `P0-01` 注册登录       | 数据库无该手机号                                  | `POST /api/auth/register` -> `POST /api/auth/login` -> `GET /api/auth/me` | `user_account` 新增；登录不改表；`/auth/me` 只读                                                               | 注册返回 `userId/phone/nickname/isVerified/status/creditScore`；登录返回 `token+userInfo`；`/auth/me` 返回 `role=USER`                  |
| `P0-02` 实名认证       | 已登录用户，`is_verified=0`                     | `POST /api/users/verify-student`                                          | `user_account.student_no` 写入；`is_verified=1`                                                        | 返回 `userId/studentNo/isVerified`                                                                                            |
| `P0-03` 创建订单       | 已实名用户                                     | `POST /api/orders`                                                        | `group_order` 新增一条 `OPEN`；`group_order_member` 新增发起人成员一条                                            | 返回 `orderId/orderNo/status/currentMemberCount/estimatedPerAmount/currentUserMember`                                         |
| `P0-04` 发起人支付      | 订单存在，发起人成员 `UNPAID`                       | `POST /api/orders/{orderId}/pay`                                          | 发起人成员 `pay_status=PAID`；新增一条 `capital_record(type=PAY)`                                             | 返回 `memberId/payStatus/payAmount/paidAt/orderStatus`                                                                        |
| `P0-05` 参与者加入      | 订单 `OPEN`，未满员，当前用户未加入                     | `POST /api/orders/{orderId}/join`                                         | `group_order_member` 新增一条；`group_order.current_member_count +1`                                     | 返回 `memberId/joinStatus/payStatus`                                                                                          |
| `P0-06` 最后一人支付自动成团 | 当前订单只差最后一人支付                              | `POST /api/orders/{orderId}/pay`                                          | 最后一个成员 `pay_status=PAID`；新增 `PAY`；`group_order.status=GROUPED`；`receipt_upload_deadline_at=now+30m` | 返回 `orderStatus=GROUPED`                                                                                                    |
| `P0-07` 未成团超时取消    | 存在 `OPEN` 订单且 `deadline_at <= now`，有已支付成员 | 触发 `OrderScheduler.cancelExpiredOpenOrders()`                             | `group_order.status=CANCELED`；`ACTIVE->CANCELED`；已支付成员新增 `REFUND_CANCEL`；`refund_amount_total` 更新   | 后续查详情/列表能看到 `status=CANCELED`                                                                                               |
| `P0-08` 订单列表       | 已有若干订单                                    | `GET /api/orders?page=1&pageSize=10`                                      | 无写操作                                                                                                | `PageResult<OrderListItemVO>`，每项包含 `orderId/orderNo/productName/.../creatorNickname/currentUserJoined/currentUserPayStatus` |
| `P0-09` 订单详情       | 已有订单、成员和支付记录                              | `GET /api/orders/{orderId}`                                               | 无写操作                                                                                                | 返回完整 `OrderDetailVO`，11 个模块齐全                                                                                               |

### 冲突修正说明

1. `P0-07` 验收必须验证**净退款**，不是只验证订单状态变了。
2. `P0-08` 和 `P0-09` 必须使用真实聚合返回，而不是只返回占位字符串。
3. `P0-06` 必须验证 `receiptUploadDeadlineAt` 已写入，否则第五步上传凭证超时任务无法联调。

# 12. MySQL 8 建表 SQL 初稿

下面这版 SQL 是**唯一初稿**。金额统一 `DECIMAL(10,2)`，状态统一 `VARCHAR`，时间统一 `DATETIME`，包含必须的唯一约束和索引，可直接执行。

```sql
SET NAMES utf8mb4;
SET FOREIGN_KEY_CHECKS = 0;

DROP TABLE IF EXISTS operation_log;
DROP TABLE IF EXISTS notification;
DROP TABLE IF EXISTS credit_change_record;
DROP TABLE IF EXISTS capital_record;
DROP TABLE IF EXISTS complaint;
DROP TABLE IF EXISTS order_receipt;
DROP TABLE IF EXISTS group_order_member;
DROP TABLE IF EXISTS group_order;
DROP TABLE IF EXISTS admin_account;
DROP TABLE IF EXISTS user_account;

CREATE TABLE user_account (
    id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '用户主键',
    phone VARCHAR(20) NOT NULL COMMENT '手机号',
    password_hash VARCHAR(255) NOT NULL COMMENT '密码哈希',
    nickname VARCHAR(50) NOT NULL COMMENT '昵称',
    student_no VARCHAR(20) NULL COMMENT '学号',
    is_verified TINYINT(1) NOT NULL DEFAULT 0 COMMENT '是否实名',
    credit_score INT NOT NULL DEFAULT 80 COMMENT '信用分',
    status VARCHAR(32) NOT NULL DEFAULT 'NORMAL' COMMENT '账号状态',
    contact_info VARCHAR(100) NULL COMMENT '联系方式',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    PRIMARY KEY (id),
    UNIQUE KEY uk_user_account_phone (phone),
    UNIQUE KEY uk_user_account_student_no (student_no),
    KEY idx_user_account_status_created_at (status, created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='普通用户表';

CREATE TABLE admin_account (
    id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '管理员主键',
    username VARCHAR(50) NOT NULL COMMENT '管理员用户名',
    password_hash VARCHAR(255) NOT NULL COMMENT '密码哈希',
    status VARCHAR(32) NOT NULL DEFAULT 'NORMAL' COMMENT '账号状态',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    last_login_at DATETIME NULL COMMENT '最近登录时间',
    PRIMARY KEY (id),
    UNIQUE KEY uk_admin_account_username (username)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='管理员表';

CREATE TABLE group_order (
    id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '拼单主键',
    order_no VARCHAR(32) NOT NULL COMMENT '拼单编号',
    creator_user_id BIGINT UNSIGNED NOT NULL COMMENT '发起人ID',
    product_name VARCHAR(100) NOT NULL COMMENT '商品名称',
    product_desc VARCHAR(255) NULL COMMENT '商品描述',
    total_member_count INT UNSIGNED NOT NULL COMMENT '总人数',
    current_member_count INT UNSIGNED NOT NULL DEFAULT 1 COMMENT '当前有效人数',
    estimated_total_amount DECIMAL(10,2) NOT NULL COMMENT '预估总额',
    estimated_per_amount DECIMAL(10,2) NOT NULL COMMENT '预估人均',
    actual_total_amount DECIMAL(10,2) NULL COMMENT '实际总额',
    actual_per_amount DECIMAL(10,2) NULL COMMENT '实际人均',
    pickup_point VARCHAR(100) NOT NULL COMMENT '取货点',
    deadline_at DATETIME NOT NULL COMMENT '截止时间',
```

### 冲突修正说明

1. 本 SQL 初稿**保留数据库级唯一约束**，但只对最关键的幂等点和唯一点加约束。
2. `operation_log` 不做物理外键，避免 `USER/ADMIN/SYSTEM` 多态操作者造成结构僵硬。
3. `credit_change_record`、`notification` 的 `related_*` 字段只建索引，不建强外键，保留历史记录灵活性。

# 14. 后端第一批建文件清单

当前仓库还是骨架阶段，第一批文件必须按真正开工顺序创建，先建基础设施，再建认证与用户，再建订单主链路。 

## 第一批：统一基础设施

| 先建哪些类                                                                                                                         | 为什么先建                      | 建完后前端能联调哪些页面                |
| ----------------------------------------------------------------------------------------------------------------------------- | -------------------------- | --------------------------- |
| `ApiResponse` `PageResult` `BusinessException` `GlobalExceptionHandler` `ErrorCodeEnum`                                       | 没有统一响应和异常，前后端无法稳定对接        | 暂无正式页面，但后端响应格式先统一           |
| 全部状态枚举类                                                                                                                       | 状态名必须先固定，避免 Service 里乱写字符串 | 前端可以先按固定状态名做映射              |
| `JwtUtil` `PasswordUtil` `CurrentLoginUtil` `OrderNoGenerator` `BizNoGenerator` `BigDecimalUtil` `MaskUtil` `TimeUtil`        | 认证、编号、金额、脱敏是后续所有业务基础       | 暂无                          |
| `JwtProperties` `StorageProperties` `JacksonConfig` `MyBatisPlusConfig` `CorsConfig` `ResourceMappingConfig` `SecurityConfig` | 没有配置类，应用无法稳定启动             | 前端可以先保证 axios 请求不会被 CORS 卡住 |
| `JwtAuthenticationFilter` `LoginIdentity` `RestAuthenticationEntryPoint` `RestAccessDeniedHandler`                            | 认证链路必须先打通                  | 登录态恢复逻辑可开始联调                |
| 10 个 `Entity` + 10 个 `Mapper`                                                                                                 | 先有数据模型，后续 Service 才能落地     | 暂无                          |
| 建表 SQL                                                                                                                        | 后端所有模块的唯一数据基线              | 暂无                          |

# 15. 前后端对接风险清单

下面这 10 个点是联调最容易炸的地方，后端必须按最终预防规则执行，不留模糊空间。

| 风险点                 | 触发场景                                                                          | 后果                         | 最终预防规则                                                                                |
| ------------------- | ----------------------------------------------------------------------------- | -------------------------- | ------------------------------------------------------------------------------------- |
| 字段名不一致              | 后端返回 `groupOrderId`，前端取 `orderId`；后端返回 `productDescription`，前端取 `productDesc` | 页面数据空白、按钮条件错乱              | **所有 JSON 字段统一 lowerCamelCase，且只用冻结契约字段名；Entity 可以 `groupOrderId`，VO 一律输出 `orderId`** |
| 时间格式不一致             | 后端直接返回 `LocalDateTime` 默认序列化或带时区                                              | 前端显示异常、表单回填失败              | **DTO/VO 的时间字段统一用 `String`，格式固定 `yyyy-MM-dd HH:mm:ss`**                               |
| 金额精度错误              | 后端用 `double` 或不统一四舍五入                                                         | 列表金额、退款金额、结算金额出现 11.999999 | **金额计算只用 `BigDecimal`，统一 `HALF_UP` 保留 2 位**                                           |
| 状态机判断不一致            | 前端以为发起人可退出；后端放行；或后端允许未实名访问订单                                                  | 联调时按钮和接口行为相互打架             | **状态名只认冻结枚举；发起人不可退出；需要实名的接口后端也必须硬校验**                                                 |
| 分页结构不一致             | 后端返回 MyBatis 默认分页对象                                                           | 前端表格不认                     | **分页响应只允许 `{ list, page, pageSize, total, pages }`**                                  |
| `actionFlags` 推导不一致 | 前端自己推导按钮，后端也推导，条件不同                                                           | 详情页按钮错显/误禁用                | **按钮展示唯一信源是后端 `actionFlags`；前端不再自行推断业务权限**                                            |
| 错误码不一致              | 后端把重复投诉写成 `40001`，前端按 `40910` 处理                                              | 前端跳错页、提示错、刷新错              | **所有业务冲突码固定按本契约实现，不能临时换码**                                                            |
| 用户 / 管理员 token 混用   | 用户 token 去调 `/api/admin/**`；管理员 token 去调 `/api/orders/**`                     | 越权或错误信息不清晰                 | **JWT 必带 `role`；权限不匹配统一返回 `40301`；`/api/auth/me` 根据 token role 查对应表**                 |
| 退款重复生成              | 用户退出、管理员取消、投诉处理、定时任务同时命中                                                      | 一笔订单被退两次                   | **所有退款与结算都必须走 `biz_no` 幂等校验；取消退款始终按 `payAmount - refundAmountTotal` 计算净额**            |
| 自动任务与手动操作并发冲突       | 用户刚上传凭证，任务同一刻扫描“未上传凭证超时”；用户刚确认收货，任务同刻自动确认                                     | 状态回滚、重复日志、重复结算             | **所有任务进入 Service 前先按当前最新数据库状态二次校验；所有关键动作走事务；任务逻辑必须幂等**                                |

### 补充风险 11：凭证图片访问失败

| 风险点               | 触发场景                                            | 后果        | 最终预防规则                                            |
| ----------------- | ----------------------------------------------- | --------- | ------------------------------------------------- |
| 凭证图片返回 URL 但前端打不开 | `/uploads/**` 被 Security 拦截或 ResourceMapping 未配 | 详情页图片加载失败 | **后端统一返回可直接访问的 `imageUrl`，并放行 `GET /uploads/**`** |

### 补充风险 12：信用分重复扣减

| 风险点                                                          | 触发场景           | 后果                | 最终预防规则                                                             |
| ------------------------------------------------------------ | -------------- | ----------------- | ------------------------------------------------------------------ |
| 投诉成立时同时写 `COMPLAINT_CONFIRMED -10` 和 `NO_RECEIPT_UPLOAD -10` | 发起人一次投诉被扣 20 分 | 前端信用分记录和后台处理结果不一致 | **V1 只保留 `COMPLAINT_CONFIRMED -10`；不再追加 `NO_RECEIPT_UPLOAD` 额外扣分** |
